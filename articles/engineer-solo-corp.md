合同会社コラレイトデザインを立ち上げてから、しばらく経ちました。フリーランスエンジニアとして受託開発をしていた自分が、なぜ法人という形を選んだのか。そして実際に運営してみて、事前の想定とどれだけ違ったのか。

この記事では「エンジニアが法人化を検討するときに読みたかった記事」を目指して、リアルな5つの学びをまとめます。手続きの解説記事はすでに良いものがたくさんあるので、ここでは**運営フェーズで気づいたこと**に絞ります。

なお、ここに書く内容はあくまで私個人の体験です。税務・法務の判断は必ず専門家にご相談ください。

---

## 1. 売上構造は「安定」と「挑戦」の2層にしないと、法人にした意味が薄れる

フリーランス時代は、来た案件を淡々とこなすだけでした。売上は受託一本、単価と稼働時間がほぼそのまま収入になるシンプルな構造です。

法人化して最初に気づいたのは、**「受託だけを続けるなら、法人にした旨味は税務面だけになってしまう」** という事実でした。それはそれで意味があるのですが、なぜわざわざ会社という器を持ったのかを考えると、もう少し先の話をしたかったはずです。

そこで意識するようにしたのが、売上の2層構造です。

- **安定層**: 受託開発・システム保守（キャッシュフローの安定化）
- **挑戦層**: 自社プロダクト・情報発信・コンテンツ（将来の資産形成）

最初のうちは受託が8〜9割を占めていて構いません。ただ、挑戦層を意識的に育てていかないと、法人という器に実態が伴わないまま維持コストだけがかかる状態になります。

フリーランスと1人法人の本質的な違いは、**「自分の時間を売る」から「会社の事業を育てる」への転換**だと思います。受託収入はその事業を育てるための燃料という位置づけにすると、毎月の案件選びに対する視点が変わってきます。

## 2. 経理は自動化できる。エンジニアにとってむしろ得意分野だった

法人化を躊躇う理由として「経理・税務が大変そう」は非常に多いと思います。自分もそうでした。

実際に運営してみると、**経理の9割は「繰り返し作業」と「データ連携」** で成り立っていました。freeeを軸に、銀行口座・クレジットカードの自動連携を設定してしまえば、日常的な仕訳の大半は自動化できます。

さらにエンジニアとして踏み込んだのが、freee APIとCloud Runを組み合わせた自動化です。請求書の作成、入金確認、月次サマリーの生成といった作業をAPIで処理するようにしてから、月の経理作業は体感で2時間程度に収まるようになりました。

```mermaid
graph LR
    A[銀行・カード明細] -->|自動同期| B[freee]
    B -->|API| C[Cloud Run]
    C -->|自動仕訳・確認| D[Discord通知]
    D -->|確認のみ| E[税理士連携]
    E -->|年次| F[確定申告]
```

「経理が怖い」と思っているエンジニアに伝えたいのは、**経理はAPIとデータパイプラインの設計とほぼ同じ思考で最適化できる**ということです。一度設計してしまえば、後はほぼ自走します。

技術的な実装の詳細は、別記事「[freee APIをCloud Runで動かして経理作業を月2時間に減らした話](/articles/freee-api-cloud-run)」で書いているので、興味のある方はそちらを参照してください。

## 3. 案件を断る基準を持つことが、最大の武器になった

フリーランス時代の自分は「依頼が来たら断りにくい」という状態でした。単価が低くても、稼働が読めなくても、なんとなく引き受けてしまう。断ることへの罪悪感のようなものがありました。

法人化してから設けたのが、案件選びの3つの基準です。

1. **粗利率35%以上**: 外注費や経費を差し引いた後の利益率がこれを下回る案件は原則受けない
2. **自分が実装担当にならない構造**: 自分がコードを書き続ける案件より、設計・ディレクションに軸足を置く構造を優先する
3. **持ち出し（無償稼働）は原則禁止**: 「まず動かしてみてから」「少し試しに」という依頼でも、工数が発生するなら費用を請求する

これらの基準を言語化したことで、**断りやすくなりました**。感情ではなく基準で判断できるので、罪悪感も薄れます。

なぜ35%なのかについては、FIRE（Financial Independence, Retire Early）的な発想が背景にあります。自分の時間は有限で、その時間単価を上げ続けることが資産形成に直結します。粗利率が低い案件は、時間あたりの収益効率が低い案件とほぼイコールです。

法人という器を使って「どの仕事を選ぶか」を設計することが、フリーランスとの本質的な差になると実感しています。

## 4. 1人でも「チーム」は作れる。AIがチームメイトになった

1人法人の最大の課題として語られるのが「孤独」と「チェック体制のなさ」です。これは実際にそうで、意思決定のすべてが自分に返ってくる。誰も間違いを指摘してくれない。

この問題に対して自分が取った解決策は2つです。

**1つ目はAIエージェントをチームメイトにすること。**

Claude Codeを中心としたAIエージェント体制を業務に組み込み、コードレビュー・記事レビュー・意思決定の壁打ちを日常的に行っています。特に「デビルズアドボケイト（DA）」ロールを持つエージェントに批判的なレビューをさせる仕組みは、1人運営の盲点を減らすのに効果的でした。

**2つ目はDiscordを業務ハブにすること。**

Cloud RunからのAPIログ、freeeの入金通知、BigQueryの集計結果、GitHubのPR情報をすべてDiscordに集約しています。通知が来たらDiscordで確認して承認するだけ、という業務フローにしたことで、「何かを見落とす」リスクが下がりました。

これらの仕組みを整えると、**「1人だけど1人じゃない」という感覚** が生まれてきます。チームメンバーの代わりに、情報とツールが自分をサポートしてくれるイメージです。

AIチームの詳しい運用については「[デビルズアドボケイトをAIチームに入れるべき理由](/articles/devils-advocate-ai-team)」、Discord業務ハブの構築については「[1人法人がDiscord Botを業務ツールにしたら想像以上に便利だった話](/articles/discord-bot-business)」を参照してください。

## 5. 法人は「ゴール設計」が9割。何のために法人にするかを問い直し続ける

これが5つの学びの中で、最も重要だと感じているものです。

法人化を検討するとき、多くの場合「節税できる」「信用が上がる」「経費の幅が広がる」といったメリットが理由に挙げられます。どれも事実ですが、**それは手段の話であって、ゴールではありません。**

自分の場合、法人化のゴールはFIREの加速です。「労働収入への依存を減らし、資産や仕組みからの収益を増やす」という方向に向かって、法人という器を使うことを選びました。

このゴールが明確になってから、日々の意思決定が楽になりました。

- 単価を上げることより粗利率を上げることを優先する
- 売上を最大化するより、自分の稼働時間を最小化することを優先する
- 法人の維持コストをできる限り低く保つ（現在の業務基盤は月額5ドル程度）

「売上を追う」だけでなく「いくらで十分か」を定義することで、案件を選ぶ基準も、投資する技術も、優先する業務も変わってきます。

法人化を考えているエンジニアの方には、手続きの前に一度「自分のゴールは何か」を言語化することをお勧めします。税理士に相談するより前に、この問いと向き合うことが一番重要だと思っています。

GCPを活用した低コスト業務基盤の詳細は「[1人法人のGCP業務基盤を月額5ドル以下で構築した話](/articles/solo-corp-gcp)」で書いています。

## まとめ: 法人は手段であり、目的ではない

5つの学びを振り返ると、共通しているのは「**法人化は何かを実現するための手段に過ぎない**」という認識です。

| 学び | 一言まとめ |
|------|-----------|
| 1. 売上の2層構造 | 受託で安定させながら、自社事業を育てる |
| 2. 経理の自動化 | エンジニアスキルで月2時間まで削れる |
| 3. 案件の断る基準 | 基準があれば感情でなく論理で動ける |
| 4. AIチームメイト | 1人でもチェック体制は作れる |
| 5. ゴール設計 | 「何のために」を問い続けることが全ての判断軸になる |

法人化を検討しているエンジニアの方に、この記事が少しでも参考になれば幸いです。疑問や共感があれば、コメントで教えてもらえると嬉しいです。
