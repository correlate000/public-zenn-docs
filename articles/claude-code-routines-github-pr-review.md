2026年4月14日、AnthropicはClaude Code Routinesをリリースした。「ローカルマシン不要でClaudeがクラウド上で自律実行する」機能だ。

リリース翌日にGitHub PR自動レビューを実装したので、その記録を書く。既存の解説記事が触れていない「プロンプトをどう書くか」「RemoteTrigger APIの実挙動」「GitHub ActionsとRoutinesどちらを使うべきか」を中心に据える。

## Routinesとは（30秒で）

3種類のトリガーで起動する自動実行の仕組み。

| トリガー | 用途 |
|---------|------|
| スケジュール | cron式で定期実行（最短1時間間隔） |
| GitHubイベント | PR opened/merged、Release publishedなど |
| API | `POST /v1/code/triggers/{id}/run` で外部から起動 |

実行はAnthropicのクラウドインフラ上。ローカルマシンがスリープしていても動く。

**プラン別の実行上限:**
- Pro: 5回/日
- Max: 15回/日
- Team/Enterprise: 25回/日

---

## 実装した内容

自社のコーヒーメディアサイト（Next.js + MDX、ja/enバイリンガル）のGitHub PRに対して、以下を自動チェックしてコメントを投稿するRoutineを作った。

1. **ja/en両言語チェック**: `content/ja/`のファイルに対応する`content/en/`が存在するか
2. **AffiliateBoxチェック**: 新規.mdxファイルに`<AffiliateBox>`コンポーネントがあるか
3. **ASINチェック**: Amazon ASINが10文字英数字の正しい形式か（AI生成の架空ASINは70%以上）
4. **revalidateチェック**: `export const revalidate`が86400（24h）になっているか

実際の出力がこれ:

```
### coffee-guide PR チェック結果
| チェック項目 | 結果 | 備考 |
|------------|------|------|
| ja/en 両言語 | 該当なし | コンテンツファイルなし |
| AffiliateBox | 該当なし | .mdx ファイルなし |
| ASIN | 該当なし | .mdx ファイルなし |
| revalidate | 該当なし | ページファイルなし |

All checks passed ✅

---
_Generated by Claude Code_
```

## Routineの作り方

### Web UIで作る（推奨）

`claude.ai/code/routines` → New routine。以下を設定する:

- **手順（プロンプト）**: 後述
- **リポジトリ**: GitHubリポジトリを紐付け
- **トリガー**: GitHubイベント → Pull request → PRのオープン

保存すれば即日アクティブになる。

### RemoteTrigger APIで作る

Claude Code内で使えるRemoteTriggerツールを使って作成も可能。ただし **いくつかの罠がある** ので後述する。

## プロンプト設計が9割

Routinesはユーザーの承認なしで自律実行される。通常のClaude Codeセッションと違い、「どこにファイルがあるか」「何をゴールとするか」を全てプロンプト内に書き切らなければならない。

**失敗するプロンプトのパターン:**

```
# ❌ これだとRoutineが迷子になる
PRをレビューして問題があれば教えてください。
```

**機能するプロンプトの構造:**

```
# ✅ 自己完結した構造
[コンテキスト] このリポジトリは何か、どんなルールがあるか
[タスク] 何を確認するか（具体的な条件）
[出力形式] どのフォーマットでコメントを投稿するか
[終了条件] 問題なし/あり の場合に何を書くか
```

実際に使っているプロンプトを公開する:

```
Review this PR for coffee-guide.jp. Post ONE summary GitHub comment:

### coffee-guide PR チェック結果
| チェック項目 | 結果 | 備考 |
|------------|------|------|
| ja/en 両言語 | ✅/❌ | |
| AffiliateBox | ✅/❌/該当なし | |
| ASIN | ✅/❌/該当なし | |
| revalidate | ✅/❌/該当なし | |

Checks:
1. ja/en: content/ja/ changes must have content/en/ counterpart
2. AffiliateBox: new .mdx files (except legal/) need <AffiliateBox>
3. ASIN: asin values must be 10-char alphanumeric like B07XJ8C8F5
4. revalidate: export const revalidate must be 86400, not 3600

End with All checks passed ✅ or list file paths needing fixes.
```

**設計のポイント:**

- **出力フォーマットをプロンプトに埋め込む**: Claudeはフォーマットの模倣が得意なので、期待するテーブルをそのまま貼る
- **英語で書く**: Routineのプロンプトは英語の方が精度が安定する（出力の日本語部分は日本語のまま維持される）
- **チェック条件は番号付きリスト**: 曖昧な自然言語より「1. 条件 2. 条件」の形式の方が漏れが減る
- **「1件だけ投稿」を明示**: POST ONE summary comment と書かないと複数コメントが投稿されることがある

## RemoteTrigger APIの実挙動（ハマりポイント）

Claude Code内でRoutineをAPIから作成しようとすると、ドキュメントには載っていない罠がある。

### `session_request` は外部非公開

公式ドキュメントには `session_request` フィールドの記述があるが、外部APIからは使えない。

```json
// ❌ これはHTTP 400になる
{
  "name": "test",
  "session_request": {
    "prompt": "..."
  }
}
// → "session_request.prompt: Extra inputs are not permitted"
```

### 正しい構造は `job_config.ccr`

実際に動く構造はこれ:

```json
{
  "name": "coffee-guide PR Review",
  "job_config": {
    "ccr": {
      "environment_id": "env_01XXXXXXXXXXXXXXXX",
      "session_context": {
        "model": "claude-sonnet-4-6",
        "sources": [
          {
            "git_repository": {
              "url": "https://github.com/your-org/your-repo"
            }
          }
        ]
      },
      "events": [
        {
          "data": {
            "uuid": "a3f7c21e-084b-4d9e-b6a1-5e2c9f0d3b87",
            "type": "user",
            "message": {
              "role": "user",
              "content": "ここにプロンプト"
            }
          }
        }
      ]
    }
  },
  "trigger": {
    "type": "github",
    "event": "pull_request",
    "actions": ["opened", "synchronize"]
  }
}
```

`environment_id` は自分のアカウントのものが必要。`claude.ai/code/routines` でRoutineを一度作ると、レスポンスJSONに含まれている。

### GitHub triggerはWeb UIで設定する方が確実

APIで `trigger` フィールドを渡してもレスポンスに反映されないことがある。GitHubトリガーの設定は Web UIから行う方が確実。

## GitHub Actions vs Routines、どちらを使うべきか

両方検討したので比較する。

| 観点 | GitHub Actions | Claude Code Routines |
|------|--------------|---------------------|
| セットアップ | workflow YMLを書く + Secrets設定 | Web UIで5分 |
| トークン管理 | ANTHROPIC_API_KEYをSecretに | Routineに紐付いた専用トークン |
| 実行コスト | GitHub Actions分（無料枠あり） | Maxなら15回/日の消費 |
| プロンプトの管理 | YMLにインライン or 別ファイル | Web UIで直接編集 |
| デバッグ | Actions logを見る | claude.ai/code/routinesのlog |
| 制限 | なし（実行回数） | 15回/日（Max） |

**Routinesを選ぶケース:**
- PRの頻度が低い（1日10本以下）
- セットアップを最小限にしたい
- Anthropicのエコシステムに統一したい

**GitHub Actionsを選ぶケース:**
- PRが大量に発生する（25回/日を超えそう）
- 既存のCI/CDパイプラインと統合したい
- 細かな条件分岐が必要

自社の場合はPR数が少ないのでRoutinesを選択した。

## 実際に使ってみた所感

**良かった点:**
- 設定が本当に5分で終わった
- プロンプトの修正がWeb UIから即日反映される
- 「該当なし」の判定が賢い（差分にMDXがなければAffiliateBoxは「該当なし」と正しく判定）

**気になった点:**
- 初回実行まで数分かかる（PRを開いてからコメントが来るまで2〜3分）
- エラー時のログがWeb UIで確認しにくい
- プロンプトのバージョン管理ができない（Gitで管理できない）

プロンプトのバージョン管理については、実運用では重要な変更のときだけメモを残す程度で対処している。

---

## まとめ

Claude Code Routinesは「PRをopenするたびに特定のチェックを自動化したい」というユースケースに非常にフィットした。

設定の本質はプロンプト設計だ。**自己完結していること**、 **出力フォーマットを明示していること**、 **終了条件が明確であること** の3つを満たすと安定して動く。

API経由での作成は `job_config.ccr` 構造を使う。GitHubトリガーの設定はWeb UI経由が確実。

MaxプランであればPR頻度が高くなければ運用上の問題はない。まず試してみる価値がある。
