LLMで記事を量産できる時代になった。しかし「生成できる」と「本番に出せる」のあいだには深い溝がある。架空の組織名、出典のない統計、非弁行為スレスレの法律解説、片方に矛盾する数値。これらはすべて、GPT・Claude の生成物に実際に混入してきた問題だ。

本稿では、2026年4月に実運用で確立した **content-validate → fact-check → DA review** の3段階品質パイプラインを解説する。4記事を同時に初稿から A- 相当まで引き上げた実例をもとに、各段の役割分担と検出できる問題のカテゴリを具体的に示す。

## なぜ1段では足りないのか

LLM生成コンテンツの品質問題は「検出できる問題の種類」によって難易度が大きく異なる。

| 問題カテゴリ | 例 | 検出に必要なもの |
|---|---|---|
| 構造・スタイル違反 | h1重複、太字の構文ミス | ルールベースの正規表現 |
| 事実の誤り | 404 URL、架空組織名 | Web検索 + HTTP確認 |
| 論理の穴、法的リスク | 非弁行為、矛盾する数値断言 | 深い批評・専門知識 |

3つの問題カテゴリは、それぞれまったく異なる検出手段を必要とする。1回のレビューで全部を担わせると、各観点が中途半端になる。また、高コストなモデル（opus）をすべての工程に投入するのはコスト効率が悪い。

3段に分けることで、各段のモデルが専門性を発揮し、opus の費用を **DA review だけに絞れる** という設計になっている。

## 全体フロー

```
初稿（LLM生成）
    │
    ▼
[Step 1] content-validate（sonnet）
    │  ルールベース: 構造・スタイル・コンポーネント整合
    │  自動修正可能な問題を排除
    │
    ▼
[Step 2] fact-check（sonnet + WebSearch）
    │  事実検証: URL到達性・組織実在・数値出典
    │  架空情報を修正または削除
    │
    ▼
[Step 3] DA review（opus Skeptic）
    │  深い批評: 論理整合・法務境界・読者リスク
    │  HIGH/MEDIUM/LOW で指摘分類
    │
    ▼
本番公開
```

スキル定義のヘッダには `model: sonnet` / `model: opus` を明記し、各段で使うモデルを固定している。

```yaml
# content-validate/SKILL.md
name: content-validate
model: sonnet

# da-review/SKILL.md
name: da-review
model: opus
```

## Step 1: content-validate — 機械的な構造チェック

### 役割

**「既知パターンの違反を全件洗い出す」** 専門スキル。DAレビューで繰り返し検出されてきたパターンをルール化し、sonnet が機械的に消化する。

### 主なチェックカテゴリ

**CAT-A: テキスト品質**

- `A-6`: frontmatter の `title` と本文 h1 の重複
- `A-7/A-8`: 太字（`**`）の構文ミス（奇数個、テーブル内太字、日本語直後スペース欠落）
- `A-11/A-12`: Citation 数の不足（3件未満でHIGH）
- `A-12.6`: 数値パターン（XX%、XX万人、XX兆円）が `<StatRef>` で囲まれていない
- `A-15`: 文末パターン3連続（です・ます・だ が3連続でMEDIUM）

**CAT-B: コンポーネント整合性**

- `B-1`: 未登録コンポーネント名（CRITICAL）
- `B-2`: locale prop の対応漏れ

**CAT-I: ja/en 整合性**

- 対になる英語版の存在確認
- コンポーネント数の一致確認

### スキルの呼び出し方

```bash
# レポートのみ
/content-validate path/to/article.mdx

# 自動修正も実行
/content-validate path/to/article.mdx --fix

# 英語版も全チェック対象に含める
/content-validate path/to/article.mdx --en
```

### 実例: 初稿に混入していた構造違反

今回の4記事では、content-validate で以下が検出された。

- `A-6` h1重複: 2件（frontmatter title と本文冒頭の見出しが一致）
- `A-7` 太字スペース欠落: 複数行（`**太字**が` → `**太字** が` への修正）
- `A-12` Citation 不足: 1件（2件のみで終わっていた記事）
- `A-12.6` StatRef 未付与の数値: 各記事に3〜7件

`--fix` オプションで自動修正できる項目（h1削除、スペース挿入）を適用し、StatRef が必要な数値については一覧を出力して次段の fact-check に引き渡す。

content-validate の目的は初稿を **B+ から A- 手前まで引き上げる** ことで、内容の正確性や論理的整合性には踏み込まない。

## Step 2: fact-check — 事実検証

### 役割

**「実在しない情報を記事から排除する」** 専門スキル。sonnet に WebSearch を組み合わせ、本文の主張を外部ソースで検証する。

### 主なチェック項目

**URL到達性確認**

```
本文中の全リンクに HTTP GET を送信。
- 200 OK → PASS
- 404 / リダイレクト先が無関係 → HIGH
- タイムアウト → MEDIUM
```

**架空組織名の検出**

LLMは存在しない団体名や研究機関名を生成することがある。たとえば「一般社団法人フリーランス支援機構」のような名称は、そのままでは実在するか判断できない。fact-check では：

1. 組織名を抽出
2. WebSearch で公式サイト・登記情報を確認
3. 実在が確認できなければ HIGH として削除推奨

**Generic URL の検出**

Webサイトのトップページ（`https://example.go.jp/`）を Citation に使っている場合、読者が情報源を検証できない。`/` で終わるURL、または1階層のみのURLを検出してMEDIUM報告する。

**表形式数値の出典逆引き**

今回の運用で明確になった fact-check の盲点がこれだ。記事末尾の Citation 一覧は存在していても、テーブル内の具体数値がどの Citation と対応するかの1対1整合を確認していなかった。

たとえば「駆け出し: 0.5〜1.0円 / 中堅: 1.5〜3.0円 / 専門: 3.0〜8.0円」という文字単価テーブルがあり、Citation に「フリーランス白書2024」が並んでいても、白書にはWebライターの文字単価細分化データは掲載されていない。この場合テーブルは著者の肌感覚であり、出典付きの断言として提示することは問題だ。

改善後の fact-check では「記事内の全数値について、対応する出典 Citation を1対1で逆引きできるか」をチェック項目に追加している。

### 実例: 今回の4記事で検出・修正した事実エラー

- 架空組織名 2件（修正: 削除 or 実在組織に差し替え）
- 404 URL 4件（修正: アーカイブURL or 削除）
- Generic URL（トップページのみ）2件（修正: 該当ページへのディープリンクに差し替え）
- 表形式数値の出典不整合は fact-check では見逃され、DA review で HIGH として検出された

fact-check 後の状態がグレードとして A- 相当になる。ここまでで「実在しない情報」は排除できている。しかし「論理的に問題のある情報」はまだ残る。

## Step 3: DA review — 論理と境界の批評

### 役割

**「内容の深い批評と法的・倫理的リスクの検出」** 専門スキル。opus を Skeptic ペルソナで動かし、HIGH/MEDIUM/LOW の3段階で指摘を出力する。

Skeptic ペルソナとは「ユーザーの損害につながる可能性のある主張を積極的に疑う批評者」の役割だ。称賛ではなく、問題の列挙が仕事になる。

### グレード判定基準

```
A   : HIGH 0件
A-  : HIGH 1件以内（軽微かつ修正容易）
B+  : HIGH 2〜3件
B   : HIGH 4件以上 or 構成に重大な問題
C以下: 再生成推奨
```

HIGH が1件でも存在すれば A- 未達。本番公開は HIGH 0件が前提条件だ。

### 実例: 今回の4記事で検出された HIGH 4件

**H-1: 文字単価テーブルが出典なし断言調**

fact-check を通過していたが、DA review で問題が発覚した。「駆け出し 0.5〜1.0円」「中堅 1.5〜3.0円」といった数値は、末尾 Citation のどの文書にも該当データが存在しない。読者がこの数値を単価交渉の根拠として使うと実害が出る可能性がある。

修正方針は3択で提示した。

- オプションA（推奨）: 具体数値を削除し、段階の「特徴・移行条件」だけを残す
- オプションB: 「著者が実務で観測した範囲であり公的統計ではない」と明記して数値を維持
- オプションC: 白書データを無理に文字単価換算せず、白書の分布データをそのまま引用

**H-2: 日当単価テーブルが出典なし断言調**

「コーダー 30,000〜60,000円 / デザイナー 40,000〜80,000円」の日当テーブルが Citation なしで提示されていた。発注者がこの数値を交渉材料にすると、受託者への不当な値下げ圧力の根拠になりかねない。

修正: テーブル直前に「編集部が案件現場で観測した目安であり、公的統計ではない」と明記。

**H-3: 税務記述が税理士法2条境界**

「独立2年目の税負担は独立初年度の売上ベースで計算されて翌年に一括請求される」という記述が問題だった。

住民税は前年所得ベース、国民健康保険も前年所得ベース、所得税は当年所得ベースで、支払い時期も制度ごとにバラバラ（住民税は4期、国保は8期前後、所得税は3月）。「翌年一括請求」は事実として不正確かつ、税理士法2条の「税務相談」に抵触するリスクがある。

修正: 「複数制度がそれぞれ異なる時期に請求される」という表現に書き換え。さらに記事冒頭に「本記事は一般情報であり税務アドバイスではない。具体判断は税理士に相談すること」を追加。この1文が HIGH を中和する安全装置になる。

**H-4: 契約条項の完成形提示が弁護士法72条境界**

SNS炎上への対処を解説する記事で「発注者の指示に基づき、発注者の事前承認を経た表現物に関する第三者からの苦情・法的請求への対応責任は、発注者に帰属する」という条項文言が完成形で提示されていた。

条項の完成形を提示することは弁護士法72条（非弁行為）との境界線上にある。また、この条項には「ただし受託者の故意・重過失の場合を除く」の但し書きがなく、受託者に不利な解釈を許容する可能性があった。

修正: 「こうした内容を弁護士に依頼して条項化する」という説明に変更し、条項文言の直前に「本記事は法的アドバイスではない。条項文言は弁護士に依頼すること」を明記。

### DA review の出力フォーマット

```
[HIGH] カテゴリ: 指摘内容
  現状: 問題のある記述の引用（行番号付き）
  問題: なぜ問題か（法的リスク・読者への実害・事実の誤り）
  推奨: 修正オプション（複数提示）
```

指摘は必ず「現状（Before）」と「推奨（After）」をセットで書く。批評だけで修正案がないレポートは実用性が低い。

### DA review 後の保存とマーカー

レビュー結果は `~/dev/Obsidian/03_knowledge/patterns/da-review-YYYY-MM-DD-{slug}.md` に保存する。また、MDX ファイルが対象の場合は pre-commit フックのゲートとして `write-quality-marker.sh` でマーカーを生成する。

```bash
~/.claude/hooks/lib/write-quality-marker.sh da-reviewed FILE1 FILE2
```

マーカーにはファイルの SHA256 ハッシュが含まれ、レビュー後に内容を変更すると再レビューが必要になる。content-validate + fact-check + da-review の3つ全てのマーカーが揃わないと commit できない設計だ。

## パイプラインの全体成績

今回の4記事への適用結果をまとめる。

| 段階 | 検出・修正内容 | グレード変化 |
|---|---|---|
| 初稿 | LLM生成のまま | B 相当 |
| content-validate 後 | 構造違反・太字ミス・Citation不足を修正 | B+ 相当 |
| fact-check 後 | 架空組織名2件・404 URL4件・Generic URL2件を修正 | A- 手前 |
| DA review 後（HIGH修正前） | HIGH 4件を検出 | B+（A- 未達） |
| HIGH修正後 | 数値断言緩和・免責注記追加・条項文言変更 | A- 到達 |

初稿から最終稿まで、HIGH 4件の修正は1時間以内で完了した。DAレビューレポートが「現状 / 問題 / 推奨」の形式で具体的な修正案を出していたため、迷わず実行できた。

## どの段階を省略すると何が起きるか

実運用での失敗パターンから、省略時のリスクを整理する。

**content-validate を省略した場合**

DA review のコンテキストに構造的なノイズが混入する。h1重複や太字の構文ミスは DA review でも指摘されるが、opus の処理コストが高い分だけ無駄が大きい。また `--fix` による自動修正の機会を失うため、同じ修正を手作業でやることになる。

**fact-check を省略した場合**

DA review は「記事が言っていること」を正しいと仮定して批評する。外部情報の確認を DA review に求めると、WebSearch が必要になり review が長大化する上、opus に検索コストも乗る。404 URLや架空組織名は fact-check の専門領域で、DA review にはこれを求めない設計にしている。

今回の実例では、fact-check が「表形式数値の出典逆引き」まで踏み込んでいなかったため、H-1・H-2 が DA review まで残存した。fact-check プロトコルにチェック項目を追加することで、次回以降は早い段階で検出できる。

**DA review を省略した場合**

法務境界（弁護士法72条・税理士法2条）と、論理的矛盾（本文内の数値の矛盾、シナリオ設定とスコープの不一致）が検出されないまま公開される。単なる品質問題ではなく、読者に実害を与えるリスクだ。「出典なし断言の相場表を信じて単価交渉した」「契約条項をそのまま使って弁護士から否定された」といったシナリオが現実に起こりうる。

## パイプラインの改善サイクル

3段パイプラインは「固定されたルールセット」ではなく、実運用のフィードバックを受けて継続的に更新される設計にしている。

今回の DA review レポートには「再発防止の知見」セクションを設け、パイプライン自体の改善候補を記録した。

1. fact-check プロトコルの改善: 「表形式の具体数値が末尾 Citation のどれと対応するか」の1対1逆引きチェックを追加
2. 法務・税務記述の安全装置ルール化: 税務・法務・契約に踏み込む記事には、記事冒頭または該当セクション冒頭に免責注記を必須化
3. 内部リンク密度の品質ゲート: 新規記事公開前に「カテゴリ内関連記事 X 件への内部リンクがあるか」を content-validate に追加

改善候補は `~/.claude/knowledge/article-quality-checklist.md` に項目を追加し、次回の content-validate から自動的に適用される。

このフィードバックループが回ることで、パイプライン全体の検出精度が徐々に上がっていく。

## まとめ

LLM生成記事を本番品質に引き上げる3段パイプラインの要点を整理する。

**段階の役割分担**

- content-validate（sonnet）: ルールベースの構造・スタイルチェック。自動修正可能
- fact-check（sonnet + WebSearch）: URL・組織名・数値出典の事実検証
- DA review（opus Skeptic）: 論理整合・法的境界・読者リスクの深い批評

**モデル選択の設計意図**

HIGH/MEDIUM/LOW の判定はルールベースと WebSearch で sonnet が担い、深い推論が必要な DA review だけに opus を投入する。コストと精度のバランスを、段階ごとに最適化している。

**省略できない理由**

各段は「前の段が見落とす問題カテゴリ」を専門に担当している。省略は「そのカテゴリの問題が検出されない」ことを意味し、法的リスクや読者への実害につながる可能性がある。

**パイプラインの自己改善**

DA review のレポートに「再発防止の知見」を含め、発見した盲点を次回の content-validate / fact-check プロトコルに即時反映する。品質ゲートは実運用のフィードバックで精度が上がる仕組みになっている。

---

スキル定義（`SKILL.md`）・チェックリスト（`article-quality-checklist.md`）の詳細は本記事のスコープ外だが、重要なのはパイプラインの構造と各段が担う問題カテゴリの分離だ。手元の LLM コンテンツ生成フローに取り込む際は、自分のコンテンツタイプに合わせてチェック項目をカスタマイズしてほしい。
