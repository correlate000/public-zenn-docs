## はじめに — AIが書いたコードを誰がレビューするのか

AI支援開発が当たり前になった今、ひとつの問いに直面しています。

**「AIが書いたコードを、AIはちゃんとレビューできるのか？」**

Qodoの「State of AI code quality in 2025」レポートによると、**46%の開発者がAI出力の正確性に疑念を持っている**にもかかわらず、AIコードをそのまま使い続けているという実態があります。また「64%の開発者がAIコードの検証に、手動コーディングと同等以上の時間がかかる」とも報告されています。

私自身、Claude Code Agent Teamsを使った開発でこの問題を強く感じていました。AIは確かに高速にコードを書いてくれます。しかし「自分が書いたコードへの批判」は得意ではありません。

そこで導入したのが**デビルズアドボケイト（DA）**という役割です。

本記事では、AIチームにDAを組み込んで運用した結果、**DAラウンド1でセキュリティ問題（.envファイル権限問題16ファイル、Discord webhook権限問題）を発見し、DAラウンド2では高重要度9件・中重要度10件・低重要度4件の計23件の指摘を検出した**実体験をお伝えします。

:::message
本記事のデータはn=1の個人実体験です。一般化には注意が必要ですが、再現可能なパターンとして紹介します。
:::

## デビルズアドボケイトとは

### 元々の概念

デビルズアドボケイト（Devil's Advocate）は、カトリック教会の列聖審査から生まれた概念です。聖人候補者の列聖（カノナイゼーション）を審査する際に、あえて「反対側の意見を代弁する役割」を置くことで、審査の公正性を担保していました。

日本語で言えば「批判役」「反論係」に相当します。

### ソフトウェア開発への応用

この概念はソフトウェア開発のレビューにも応用できます。

通常のコードレビューでは、レビュアーは「このコードを承認するか否か」という判断をします。しかしDAは立場が違います。**「このコードに何が問題あるか」を徹底的に探すこと**が役割です。

「問題を見つけること」が目的なので、些細な点でも声に上げます。建設的批判を超えて、あえて厳しい目線で見ることが求められます。

### AIエージェントとしてのDA

Claude Code Agent Teamsの公式ドキュメントには、次のような記述があります。

> Use case example: "Investigate with competing hypotheses"
> teammates explicitly adversarial: each one's job is not only to investigate its own theory but to challenge the others'

つまり、**Anthropic自身がAgent Teamsの設計思想としてadversarialな（敵対的な）構造を想定している**のです。

また、ペンシルバニア大学とGoogle DeepMindの研究（DEVIL'S ADVOCATE論文）では、LLMエージェントに「先読み能力（anticipatory thinking）」を持たせるアプローチが研究されており、「失敗を予測し代替戦略を準備する能力」がエージェントの品質向上に寄与することが示されています。マルチエージェントディベート研究においても、adversarial structuresがLLMの出力品質を改善するという結果が出ています。

## 実践 — Claude Code Agent TeamsでDAを運用した

### チーム構成

私のClaude Code Agent Teamsは以下の5ロール構成です。

| ロール | 役割 |
|--------|------|
| **Pragmatist** | 実用性重視。「今すぐ動くか」を判断する |
| **Skeptic（DA）** | 批判役。問題点を徹底的に洗い出す |
| **Idealist** | 理想論。「本来どうあるべきか」を問う |
| **Connector** | 統合役。各意見を整理し結論をまとめる |
| **Implementer** | 実装担当。上記の議論を元にコードを書く |

この中で**Skeptic（デビルズアドボケイト）が最も重要**です。

CLAUDE.mdには次のルールを明記しています。

```markdown
## Agent Teams
- 4性格体制を推奨: Pragmatist / Skeptic / Idealist / Connector
- Skeptic（DA）は必須メンバー。省略禁止
- DAレビューはR1→R2の2ラウンド実施を標準とする
```

### DA Round 1: セキュリティ問題の発見

Mac mini M4 Proのセットアップ後、環境全体をDAにレビューさせました。

指示はシンプルです。

```
あなたはデビルズアドボケイト（厳格なセキュリティ監査役）です。
この開発環境のセキュリティ上の問題点を徹底的に洗い出してください。
見落としよりも過剰検出を優先してください。
```

その結果、DAが発見した主な問題は以下でした。

**発見1: .envファイルの権限問題（16ファイル）**

```bash
# DAが指摘した問題の確認
find ~/dev -name ".env*" -exec ls -la {} \; | grep -v "600"

# 問題のあるファイルが16件ヒット
# 権限が 644（誰でも読める状態）になっていた
```

```bash
# DAの指示で実行した修正
find ~/dev -name ".env*" -exec chmod 600 {} \;

# 修正後の確認
find ~/dev -name ".env*" -exec ls -la {} \;
# 全て -rw------- になったことを確認
```

.envファイルに644（所有者以外も読み取り可能）の権限が設定されていたのは、新しいMacへのデータ移行時に権限情報が失われていたためです。DAがいなければ気づかずに放置していた可能性が高いです。

**発見2: Discord webhookの権限問題**

DAはWebhook URLが環境変数ではなく設定ファイルに記述されているケースを指摘しました。Webhook URLは推測されにくいとはいえ、流出すると悪用されるリスクがあります。即座に環境変数へ移行しました。

### DA Round 2: 包括的レビュー

R1の修正を反映した後、R2として包括的なコードレビューを実施しました。

DAへの指示：

```
Round 1の修正を踏まえた上で、コードベース全体を以下の優先順位で
レビューしてください:
1. correctness（正確性）
2. error handling（エラー処理）
3. performance（パフォーマンス）
4. security（セキュリティ）
5. maintainability（保守性）
6. testing gaps（テスト不足）

各指摘は HIGH / MEDIUM / LOW の重要度で分類してください。
```

結果は以下でした。

| 重要度 | 件数 |
|--------|------|
| HIGH | 9件 |
| MEDIUM | 10件 |
| LOW | 4件 |
| **合計** | **23件** |

HIGH指摘の主な内容（抽象化して記載）：

- **エラーハンドリング不足**: Cloud Run APIの複数エンドポイントでエラー時のフォールバックがない
- **認証チェックの漏れ**: 特定のAPIエンドポイントで認証をスキップするケースがある
- **SQLインジェクション的な問題**: BigQueryクエリの動的生成箇所でのパラメータバインド不足
- **タイムアウト未設定**: 外部API呼び出しにタイムアウトが設定されていない

### 反映率35%という現実

23件の指摘のうち、即座に対応したのは**6件（35%）**でした。

残りの65%は対応しなかったわけではなく、以下のように分類しました。

```
即時対応（6件/26%）: HIGH且つ修正が1時間以内
近日対応（8件/35%）: HIGH/MEDIUMだが、影響範囲の調査が必要
保留（5件/22%）: MEDIUMだが、現状の利用パターンでは影響しない
却下（4件/17%）: LOWかつトレードオフを考慮して意図的に許容
```

**DAの指摘を全て反映する必要はありません**。DAの価値は「問題を全て修正させること」ではなく「問題を全て把握させること」にあります。把握した上で意思決定するのは人間（または他のエージェント）の役割です。

## なぜDAが効くのか — 3つの理由

### 理由1: AIは「よいしょ」しがち（確証バイアスの打破）

LLMには「ユーザーの期待に沿う方向で回答する」という傾向があります。いわゆる**sycophancy（媚び）問題**です。

「このコードに問題はありますか？」と聞くと、「概ね良いですが、細かい点では...」という回答になりがちです。しかし「このコードの問題点を徹底的に探してください」と役割を与えると、同じモデルがまったく異なる視点で分析します。

**役割設定によってLLMの出力傾向が変わる**という特性を活用するのがDAです。

### 理由2: 1人のレビュアーは偏る（多角的レビュー）

Claude Codeの公式ドキュメントには次の記述があります。

> 「1人のレビュアーは1種類の問題に偏る。3人なら互いの見落としをカバーできる」

セキュリティに詳しいエンジニアはセキュリティ問題を見つけますが、パフォーマンス問題を見落とすことがあります。DAに「correctness → error handling → performance → security → maintainability → testing gaps」という**優先順位付きの観点リスト**を渡すことで、偏りのない多角的レビューが可能になります。

Qodoのデータでも、**AIレビューあり81% vs なし55%の品質改善**という差が出ています。

### 理由3: セキュリティは攻撃者目線が必要（Red Team思考）

セキュリティ分野では「Red Team（攻撃者役）」と「Blue Team（防御者役）」を分けて演習するのが標準的な手法です。

DAはまさに**開発プロセスの中のRed Team**です。「このコードを悪用しようとしたらどうするか」「このエンドポイントを突こうとしたらどうするか」という攻撃者目線で見ることで、通常のレビューでは見つかりにくいセキュリティ問題を発見できます。

.envファイルの権限問題も、「このシステムに不正アクセスしようとしたらどこを狙うか」という視点があったからこそ発見できた問題です。

## DA運用のベストプラクティス

### 1. CLAUDE.mdへのルール明記

DAを「オプション」にすると、忙しいときに省略してしまいます。ルールとして明文化することが重要です。

```markdown
# ~/.claude/CLAUDE.md または CLAUDE.md

## Agent Teams運用ルール
- 4性格体制必須: Pragmatist / Skeptic（DA）/ Idealist / Connector
- Skepticの省略禁止。「時間がないから」でスキップ不可
- 新機能実装後のDAレビューを必須ゲートとする
- DAレビューなしでのgit commit禁止（pre-commitフックで担保）
```

### 2. DAに渡すべきプロンプト設計

効果的なDAプロンプトの要素は以下です。

```
# DAプロンプトテンプレート

あなたはデビルズアドボケイト（批判的レビュアー）です。
以下のコード/設計を徹底的に批判してください。

## あなたの役割
- 問題を「見つけること」が目的です
- 承認することは仕事ではありません
- 些細な問題でも報告してください
- 「概ね良い」という評価は禁止です

## レビュー優先順位
1. correctness（バグ・ロジックエラー）
2. error handling（例外・エラーパス）
3. performance（ボトルネック・N+1）
4. security（権限・認証・インジェクション）
5. maintainability（複雑性・命名・コメント）
6. testing gaps（テスト未カバー領域）

## 出力フォーマット
各指摘を以下の形式で記載:
- 重要度: HIGH / MEDIUM / LOW
- 場所: ファイル名:行番号
- 問題: 何が問題か（1行）
- 理由: なぜ問題か（2-3行）
- 提案: どう修正すべきか（任意）

## 対象コード
[ここにコードを貼り付け]
```

### 3. DA指摘の優先順位付け

DAの指摘を受け取ったら、以下のマトリクスで仕分けします。

```
影響度 × 発生確率 = 対応優先度

HIGH指摘:
  - 本番で問題が起きる可能性が高い → 即時対応
  - 起きたときの影響が大きい → 今週中に対応

MEDIUM指摘:
  - 技術的負債として記録 → バックログに追加
  - 次のスプリントで検討

LOW指摘:
  - 意図的に許容するか議論 → 却下 or 保留を明記
  - 「なぜ許容するか」をコメントに残す
```

重要なのは「全て修正する」ではなく「全て把握して意思決定する」ことです。

### 4. 反復レビューサイクル（R1→R2）

```
R1（初回DAレビュー）
  ↓ HIGH指摘を修正
R2（二次DAレビュー）
  ↓ 「R1で〇〇を修正しました。反映できていますか？残存問題はありますか？」
  ↓ MEDIUM以下の指摘をバックログ化
完了（またはR3へ）
```

R2ではR1からの修正内容をDAに伝えることが重要です。「修正したつもりが別の問題を生んでいた」というケースを検出できます。

## ツール紹介 — claude-devils-advocate スラッシュコマンド

独立したOSSとして、Claude Code用のDAスラッシュコマンドが公開されています。

- リポジトリ: [richiethomas/claude-devils-advocate](https://github.com/richiethomas/claude-devils-advocate)

### インストール方法

```bash
# リポジトリをクローン
git clone https://github.com/richiethomas/claude-devils-advocate.git /tmp/da-command

# スラッシュコマンドをClaude Codeのコマンドディレクトリにコピー
cp /tmp/da-command/devils-advocate.md ~/.claude/commands/devils-advocate.md

# 確認
ls ~/.claude/commands/
# devils-advocate.md が表示されればOK
```

### 使い方

```bash
# Claude Codeのチャットで実行
/devils-advocate 3    # 最大3ラウンドのレビュー
/devils-advocate 6    # 最大6ラウンドのレビュー（デフォルト）
```

### レビューの優先順位（コマンド定義より）

このコマンドは以下の順序でレビューを行います。

1. **correctness** — バグ・ロジックエラー
2. **error handling** — 例外処理・エラーパス
3. **performance** — ボトルネック・非効率
4. **security** — 権限・認証・脆弱性
5. **maintainability** — 複雑性・可読性
6. **testing gaps** — テストカバレッジ不足

### カスタマイズ

`~/.claude/commands/devils-advocate.md`を編集することで、自分のプロジェクト向けにカスタマイズできます。例えば、BigQueryを使うプロジェクトであれば「SQLインジェクション的なクエリ動的生成」を優先チェック項目に追加するといった具合です。

## よくある疑問

### Q: DAはどのモデルで実行すべきか？

コードの正確な理解が必要なため、**Claudeの場合はOpusを推奨**します。

私の環境では以下のモデル選択ガイドラインを設けています。

- **Opus**: コードレビュー・設計判断・セキュリティ監査（DAはここ）
- **Sonnet**: 探索・分析・軽量レビュー
- **Haiku**: テンプレート埋め・パース・集約

DAレビューは「間違えると手戻りが大きい」タスクなので、Opusが適切です。

### Q: DAに全部任せてよいか？

任せてはいけません。DAはあくまで「問題の発見」が役割です。

**「発見された問題をどう扱うか」は人間が判断する**必要があります。DAが「この箇所はセキュリティリスクがある」と言っても、それがビジネス上の制約から許容されているケースもあります。

Human in the Loopの原則は守ってください。

### Q: DAのレビューに時間がかかりすぎる

レビュー対象のスコープを絞ることをお勧めします。

```bash
# 変更したファイルのみをDAにレビューさせる
git diff --name-only HEAD~1 | head -20
# → 変更ファイルリストをDAに渡す
```

全コードベースを一度にレビューさせるより、「今回のPRで変更したファイルのみ」をレビューさせる方が精度も上がります。

## まとめ — 明日からDAを入れよう

デビルズアドボケイトをAIチームに組み込むことで得られるのは以下です。

1. **セキュリティ問題の早期発見**: 私の場合、.envファイル権限問題16件を発見・修正
2. **多角的なコード品質向上**: 6つの観点からの体系的なレビュー
3. **「把握して判断する」文化**: 問題を無視せず、意図的に許容/修正を選択する習慣

DAの導入コストは低いです。CLAUDE.mdにルールを1行追加し、プロンプトを設定するだけで始められます。

もし「AIが書いたコードを信頼しきれない」「一人開発でレビュアーがいない」という状況にあるなら、ぜひDAを試してみてください。

**AIに批判させることで、AIへの信頼が増す**という逆説的な体験ができるはずです。
