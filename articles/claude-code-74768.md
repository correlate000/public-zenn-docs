## なぜAIにデバッグを頼むと迷走するのか

「Claude Codeにエラーを投げたら、的外れな修正を提案されて時間を無駄にした」

この体験は、AIコーディングツールが普及した2024〜2025年にかけて急増しています。原因はツールの性能ではなく、**使い方にあります。**

デフォルト状態のClaude Codeは「補完バイアス」を持っています。エラーメッセージを受け取ると、それっぽい答えを素早く返そうとします。しかし、シニアエンジニアがデバッグ時に実際にやっていることは、まったく別の思考プロセスです。

本記事では、シニアエンジニアが無意識に実行している「仮説→切り分け→検証→絞り込み」のループを言語化し、Claude Codeに移植する具体的な方法を解説します。

## シニアエンジニアのデバッグ思考を解剖する

### 「闇雲に調べる」と「切り分けて調べる」の違い

同じエラーに直面したとき、ジュニアとシニアのアプローチは根本から異なります。

**ジュニアエンジニアの場合:**

```
エラーメッセージをコピー
→ Googleで検索
→ Stack Overflowのトップ回答を試す
→ 動かなかったら別の回答を試す
→ なんとなく動いた → 原因不明のまま前進
```

**シニアエンジニアの場合:**

```
エラーメッセージを読む
→ 「これはどのレイヤーで起きているか」を特定する
→ 「いつから、どの条件で発生するか」を確認する
→ 原因候補を3〜5個挙げる
→ 最も速く否定できる候補から検証する
→ 候補を絞り込んで原因を特定する
→ 修正 + 再発防止の設計
```

この差を一言で表すなら、**仮説を持ってから動くか、仮説なしに動くか**です。

### 仮説駆動型デバッグの5フェーズ

シニアエンジニアの思考を分解すると、5つのフェーズに整理できます。

| フェーズ | 問い | やること |
|---------|------|---------|
| **Phase 1: 症状の記述** | What？ | エラーの正確な状況を言語化する |
| **Phase 2: 再現条件の特定** | When / Where？ | いつ、どこで、どんな条件で発生するかを確認する |
| **Phase 3: 原因空間のマッピング** | Why の候補列挙 | 考えられる原因を網羅的にリストアップする |
| **Phase 4: 切り分けによる絞り込み** | How to isolate？ | 最速で否定できる候補から検証する |
| **Phase 5: 修正と再発防止** | Fix & Prevent | 原因を修正し、同じバグが出ない設計にする |

このフレームワーク自体は「知っている」人が多いですが、実際のデバッグセッションで**意識的に実行できている人は少ない**です。そして、AIはこのフレームを持っていない状態で使われることがほとんどです。

### 「切り分け」の具体技法

Phase 4の切り分けには、いくつかの定番パターンがあります。

**環境の切り分け**
```
本番でのみ発生？ → 環境差を調べる（env変数、インフラ設定）
ステージングでも発生？ → コードの問題
ローカルでも発生？ → 環境非依存のバグ
```

**時系列の切り分け**
```
いつから発生したか特定できる？
→ git log / git bisect でコミット差分を絞り込む
```

**レイヤーの切り分け**
```
DB → API → フロント → インフラ
どのレイヤーまでは正常に動作しているか？
```

**データの切り分け**
```
全ユーザーで発生？ → 普遍的なバグ
特定ユーザーのみ？ → データ起因かロール/権限の問題
特定の入力値のみ？ → バリデーション・型変換の問題
```

## Claude Code に「思考の型」を移植する

### なぜデフォルトのClaude Codeはデバッグが苦手なのか

Claude Codeがデバッグで迷走しやすい理由は3つあります。

**1. 補完バイアス**
LLMは「それらしい答えを生成する」ことに最適化されています。デバッグに必要な「わからない、もっと情報が要る」という立ち止まりが苦手です。

**2. コンテキスト不足**
AIはあなたのシステムのアーキテクチャ、デプロイ構成、過去の障害履歴を知りません。エラーメッセージだけを渡して「なぜ？」を聞いても、的外れな推測になります。

**3. 検証なき推論**
「このエラーは〇〇が原因だと思います」と即答しますが、その仮説を検証するコマンドを実行して確認するステップが抜けがちです。

これらの問題を解決するために、**「思考の型」をプロンプトとして渡す**設計が有効です。

### CLAUDE.md による思考フレームの固定

プロジェクトルートの `CLAUDE.md` にデバッグ原則を記述しておくことで、すべてのセッションに思考の型を適用できます。

```markdown
## デバッグ調査の基本原則

デバッグを依頼されたとき、以下の手順を必ず守ること。

1. **推測で回答する前に仮説リストを提示する**
   - 最低3つ、できれば5つの原因候補を列挙する
   - 各仮説には「なぜそう考えるか」の根拠を添える

2. **各仮説には「確認方法」をセットで提示する**
   - 実行可能なコマンドまたは確認すべきファイルを明示する
   - 「調べてみてください」ではなく、具体的なコマンドを書く

3. **ひとつの可能性に決め打ちしない**
   - 「原因は〇〇です」と断言する前に、他の可能性を否定したか確認する

4. **確認できていない事実と推測を明確に区別する**
   - 「確認済み:」と「推測:」を必ずラベリングする

5. **切り分けの順序を示す**
   - どの仮説を先に検証すべきか、その理由を添えて優先度を付ける
```

### デバッグ専用カスタムコマンドの設計

`.claude/commands/` ディレクトリにカスタムコマンドを作成することで、毎回プロンプトを書く手間を省けます。

```bash
# ディレクトリ構成
.claude/
└── commands/
    ├── debug-investigate.md  # 調査フェーズ
    ├── debug-bisect.md       # git bisect連携
    └── debug-report.md       # 調査結果まとめ
```

**`debug-investigate.md` の設計例:**

```markdown
# デバッグ調査コマンド

引数として渡されたエラー情報・症状に対して、以下の手順で調査を実行してください。

## Step 1: 症状の整理

まず以下を確認・整理してください：
- エラーメッセージの正確な内容
- 発生環境（本番 / ステージング / ローカル）
- 発生頻度（常時 / 特定条件 / 断続的）
- 最後に正常動作を確認した時点

不明な場合は質問してください。推測で進まないこと。

## Step 2: 原因候補のリストアップ

以下の形式で、最低5つの原因候補を提示してください：

| 優先度 | 仮説 | 根拠 | 確認コマンド |
|--------|------|------|------------|
| 高 | ... | ... | `...` |
| 中 | ... | ... | `...` |
| 低 | ... | ... | `...` |

## Step 3: 切り分け手順の提示

優先度順に、何を確認すれば候補を絞り込めるか、ロードマップを示してください。

$ARGUMENTS
```

このコマンドは `/debug-investigate [エラーメッセージ]` の形式で呼び出せます。

## 実践：実際のバグに適用してみる

### ケーススタディ①「本番だけ遅い」パフォーマンス問題

**症状:** 本番環境でAPIレスポンスが時折10秒超になる。ローカルでは再現しない。

**型なしで投げた場合（NG例）:**

```
ユーザー: 本番でAPIが遅いです。レスポンスに10秒かかることがあります。
Claude: データベースのクエリが最適化されていない可能性があります。
        INDEXを追加してみてください。また、N+1問題が発生していないか
        確認することをお勧めします。
```

問題は「可能性があります」で終わっていること。確認もしていないし、他の原因を検討する姿勢もありません。

**型を渡した場合（OK例）:**

```
ユーザー: /debug-investigate 本番環境でAPIレスポンスが時折10秒超になる。
          ステージングとローカルでは再現しない。

Claude: 【症状の整理】
        - 本番のみ発生、他環境では再現しない
        - 「時折」→ 断続的発生、常時ではない
        - この2点から、コードのバグではなく環境差・負荷・外部依存の
          問題を優先的に疑うべきです。

        【原因候補 (優先度順)】

        | 優先度 | 仮説 | 根拠 | 確認方法 |
        |--------|------|------|---------|
        | 高 | DB接続プールの枯渇 | 断続的+本番のみ | `SHOW STATUS LIKE 'Threads_connected'` |
        | 高 | 外部API呼び出しのタイムアウト | 本番と開発で異なるエンドポイント | ログでどのエンドポイントが遅いか確認 |
        | 中 | 本番のみ有効な処理（キャッシュなし等）| 環境差が原因なら設定を疑う | env変数の差分確認 |
        | 中 | GC停止（JVMやNode.jsのヒープ） | 断続的な遅延はGCで発生しうる | メモリ使用量のモニタリング |
        | 低 | ネットワークレイテンシ（CDN・ロードバランサー） | インフラ起因 | CloudWatch/Datadogのネットワークメトリクス |

        【切り分け手順】
        1. まずログでどのエンドポイントが遅いかを特定
        2. 外部API呼び出しが含まれる場合は、その応答時間を個別計測
        3. DBの接続数を確認
        4. それでも原因不明なら、APMツールで処理の内訳を確認
```

回答の質が根本的に変わります。「確認済み」と「推測」が分かれており、次のアクションが具体的です。

### ケーススタディ②「特定ユーザーだけエラー」認証問題

**症状:** 一部ユーザーのみ403エラーが発生する。

この種の問題には「データの切り分け」プロンプトが有効です。

```markdown
# 投げるプロンプトの設計

以下の切り分けを順番に実施してください。

【データの切り分け】
1. エラーが発生しているユーザーに共通する属性を探す
   - ロール（admin / user / guest）
   - 登録日（特定時期以降）
   - 地域・タイムゾーン
   - 認証方法（OAuth / パスワード / SSO）

2. 正常なユーザーとエラーユーザーの差分を確認するSQLを生成する

3. 権限テーブルのスキーマを確認して、403の原因になりうる条件を列挙する
```

Claude Codeはこの指示に従って、実際に確認用SQLを生成してくれます。

```sql
-- Claude Codeが生成した切り分けクエリ例
SELECT
  u.id,
  u.role,
  u.created_at,
  u.auth_provider,
  p.permission_flags,
  COUNT(e.id) as error_count
FROM users u
LEFT JOIN permissions p ON u.id = p.user_id
LEFT JOIN error_logs e ON u.id = e.user_id AND e.status_code = 403
GROUP BY u.id, u.role, u.created_at, u.auth_provider, p.permission_flags
HAVING error_count > 0
ORDER BY error_count DESC;
```

### ケーススタディ③「昨日まで動いていた」デグレ問題

デグレには `git bisect` との組み合わせが強力です。

```bash
# Claude Codeへの指示パターン
「直近50コミットの変更サマリーを作成して、
 認証・権限・ミドルウェアに関連する変更を特定してください。
 その後、git bisect で二分探索する手順を示してください。」
```

Claude Codeが生成する手順例:

```bash
# git bisect の実行手順（Claude Code生成）
git bisect start
git bisect bad HEAD                    # 現在のコミット（バグあり）
git bisect good abc1234                # 正常だったコミットのハッシュ

# 各ステップでテストを実行
# 正常なら: git bisect good
# バグありなら: git bisect bad

# 自動化する場合
git bisect run npm test -- --grep "auth"
```

## デバッグ自動化パイプラインの設計

### 全体アーキテクチャ

個別のデバッグを超えて、チームで再現性のある調査プロセスを構築するには、自動化パイプラインの設計が必要です。

```
[エラー発生]
    ↓
[ログ収集・整形スクリプト]
    ↓
[Claude Code: 仮説生成フェーズ]  ← 思考の型を適用
    ↓
[Claude Code: 検証コード生成フェーズ]
    ↓
[自動実行 & 結果収集]
    ↓
[Human Review Point]  ← ここを削らない
    ↓
[Claude Code: 原因特定・修正案提示]
```

**ログ収集スクリプトの例（Node.js）:**

```typescript
// scripts/collect-debug-context.ts
import { execSync } from "child_process";
import * as fs from "fs";

interface DebugContext {
  errorMessage: string;
  environment: string;
  recentCommits: string;
  recentLogs: string;
  envDiff: string;
}

function collectDebugContext(errorMessage: string): DebugContext {
  const environment = process.env.NODE_ENV ?? "unknown";

  // 直近10コミットのサマリー
  const recentCommits = execSync("git log --oneline -10").toString();

  // 直近のエラーログ（本番では Cloud Logging / DataDog から取得）
  const recentLogs = execSync(
    `grep -i "error\\|warn" /var/log/app.log | tail -50`
  ).toString();

  // ステージングと本番のenv差分（センシティブな値はマスク）
  const envDiff = getEnvDiff();

  return {
    errorMessage,
    environment,
    recentCommits,
    recentLogs,
    envDiff,
  };
}

function getEnvDiff(): string {
  // センシティブな変数をマスクして差分を出力
  const sensitiveKeys = ["SECRET", "PASSWORD", "KEY", "TOKEN"];
  const lines = Object.entries(process.env)
    .filter(([key]) => !sensitiveKeys.some((s) => key.includes(s)))
    .map(([key, val]) => `${key}=${val}`)
    .join("\n");
  return lines;
}

// 収集したコンテキストをClaude Codeへ渡すMarkdownとして出力
function generateDebugPrompt(context: DebugContext): string {
  return `
# デバッグ調査依頼

## エラー情報
\`\`\`
${context.errorMessage}
\`\`\`

## 環境
- NODE_ENV: ${context.environment}

## 直近10コミット
\`\`\`
${context.recentCommits}
\`\`\`

## 直近のエラーログ（抜粋）
\`\`\`
${context.recentLogs}
\`\`\`

## 環境変数差分（センシティブ除外）
\`\`\`
${context.envDiff}
\`\`\`

---

上記の情報をもとに、CLAUDE.mdのデバッグ原則に従って調査を開始してください。
`;
}
```

### Human-in-the-Loopの設計原則

自動化で最も重要なのは、**どこで人間が判断するか**を設計することです。

**AIに任せていい判断:**
- ログの解析と仮説の列挙
- 検証用コマンド・SQLの生成
- 原因候補の優先度付け
- ドキュメントの調査と参照

**人間のレビューが必須な判断:**
- 本番DBへのクエリ実行
- 設定変更・デプロイの承認
- 「これが原因です」という最終判断
- 修正コードのレビューとマージ

「AIが提案し、人間が承認して実行する」というフローを崩さないことが、デバッグ自動化の安全な運用に不可欠です。

### チームへの展開

CLAUDE.mdとカスタムコマンドをGitで管理することで、チーム全体に思考の型を展開できます。

```bash
# リポジトリ構成
project/
├── .claude/
│   ├── commands/
│   │   ├── debug-investigate.md
│   │   ├── debug-bisect.md
│   │   └── debug-report.md
│   └── CLAUDE.md          # デバッグ原則を記述
└── scripts/
    └── collect-debug-context.ts
```

新メンバーのオンボーディングに組み込む際は、「このコマンドを使えばOK」ではなく、**なぜこう設計したか**の背景を共有することが重要です。型を知らずに使うと、また属人化します。

## 落とし穴と限界を正直に語る

### Claude Codeが苦手なデバッグパターン

**1. 環境固有の問題**
本番サーバーへのSSHアクセスやクラウドコンソールの操作が必要なケースでは、Claude Codeは情報を取得できません。ログをコピー&ペーストするか、MCP経由でツールと接続する必要があります。

**2. ヒューリスティックが必要な問題**
「なんか動作が変」「レスポンスがぎこちない」といった定性的な症状は、AIが扱いにくいです。症状を数値化・言語化してから渡すことが前提になります。

**3. 深いドメイン知識が必要な問題**
業務ロジックの深い部分（「なぜこの計算式なのか」「この業務規則の意図は何か」）は、コンテキストを大量に渡しても精度が落ちます。

### コンテキスト汚染問題

長いデバッグセッションでは、途中で提示した誤った仮説がその後の回答に影響することがあります。

**対策:**
```bash
# 新しい仮説を立てるときは、新しいセッションを開始する
# または、明示的にコンテキストをリセットする

「先ほどまでの仮説はいったん置いて、最初から考え直してください。
 確認済みの事実: [...]
 まだ検証できていないこと: [...]」
```

### コスト設計

デバッグ自動化を本格運用する前に、トークン消費量を見積もっておく必要があります。

| シナリオ | 入力トークン目安 | 注意点 |
|---------|---------------|--------|
| ログ + エラーのみ | 2,000〜5,000 | 比較的低コスト |
| ログ + コードベース全体 | 50,000〜200,000 | Extended Contextモデルが必要 |
| 自動化パイプライン（複数回往復） | 100,000〜500,000 | コスト上限の設定を推奨 |

Extended Thinkingモードはデバッグの質を向上させますが、トークン消費が数倍になります。重要度の高いバグにのみ適用する運用が現実的です。

## まとめ：思考の型を言語化することがシニアエンジニアの仕事になる

AIコーディングツールが普及した今、シニアエンジニアの価値は**コードを書く速度ではなく、思考の型を言語化してチームとAIに渡す設計力**にシフトしています。

本記事の要点をまとめます：

- **AIのデバッグ迷走の原因は補完バイアスとコンテキスト不足**。ツールではなく使い方の問題
- **仮説駆動型デバッグの5フェーズ**（症状の記述→再現条件→原因空間マッピング→切り分け→修正&防止）を意識する
- **CLAUDE.mdでデバッグ原則を固定**することで、すべてのセッションに思考の型が適用される
- **カスタムコマンドでプロンプトテンプレート化**することで、チーム全員が同じ質の調査を実行できる
- **Human-in-the-Loopの設計を省略しない**。AIは仮説と検証コードを生成し、実行と判断は人間が行う

### 今日から使えるチェックリスト

```
□ CLAUDE.mdにデバッグ原則を追加したか？
□ .claude/commands/ にdebug-investigateコマンドを作成したか？
□ エラーを投げる前に「環境・再現条件・時系列」を整理したか？
□ AIの回答で「確認済み」と「推測」が区別されているか確認したか？
□ 本番環境への操作はAIの提案を必ず人間がレビューしているか？
```

暗黙知を言語化してAIに渡す──この設計の積み重ねが、チームのデバッグ効率を底上げします。
