## はじめに

Mac mini M4 Pro（64GB）のセットアップ作業で、私は初めてClaude Code Agent Teamsを実務に投入しました。環境構築・セキュリティ監査・ドキュメント整備を同時に進めなければならない状況で「これは並列化できる」と判断し、5エージェントチームを組んで動かしてみたのです。

このとき気づいたことが2つあります。

1つ目は、**デビルズアドボケイト（批判的レビュー役）をチームに必ず入れると、品質が劇的に変わる**こと。エージェント全員が「実装する側」でまとまってしまうと、見落としやハルシネーションの指摘が誰からも入りません。

2つ目は、**Agent Teamsが効果を発揮する前提として、CLAUDE.mdの設計が重要**だということ。各エージェントは起動時にCLAUDE.mdを読み込みます。そこに役割・制約・判断基準が整備されていないと、チームメンバーがバラバラの方向に動きはじめます。

本記事では、この2回の実践（5エージェント→4エージェント）を通じて得た知見を、設定方法から運用上の判断まで具体的に共有します。

:::message
Agent Teamsは2026年2月5日、Claude Opus 4.6と同時にリリースされたResearch Preview機能です。設定や仕様は今後変更される可能性があります。
:::

## Agent Teamsの基本構造

まず「サブエージェント」との違いを整理します。

| 観点 | サブエージェント | Agent Teams |
|---|---|---|
| エージェント数 | 1体（逐次） | 複数（並列） |
| 制御方式 | 親→子への指示 | Team lead ↔ Teammates |
| 結果の共有 | 親へ返却 | Mailbox + Task list |
| セッション再開 | 可能 | **不可**（制限あり） |
| ネスト | 可能 | **不可**（制限あり） |

Agent Teamsは4つのコンポーネントで構成されています。

- **Team lead**: チームを管理し、タスクを割り当てる役割
- **Teammates**: 実際のタスクを並列実行するエージェント
- **Task list**: 進行状況の管理（`[ACTIVE]` / `[DONE]` / `[PENDING]`）
- **Mailbox**: エージェント間のメッセージパッシング

## セットアップ

### 有効化

`~/.claude/settings.json` に以下を追加します。

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

または、セッション開始前に環境変数をエクスポートしても構いません。

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude
```

### 表示モードの選択

Agent Teamsには2つの表示モードがあります。

| モード | 要件 | 説明 |
|---|---|---|
| `in-process`（デフォルト） | なし | 1つのターミナルで全エージェントの出力を交互に表示 |
| `split-panes` | tmux または iTerm2 | 各エージェントが独立したペインで動作 |

`split-panes`モードはエージェントの動作をリアルタイムで俯瞰できるため、デバッグや状態確認がしやすいです。tmuxを使っている場合は以下で有効にできます。

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1",
    "CLAUDE_CODE_AGENT_TEAMS_DISPLAY": "split-panes"
  }
}
```

### セキュリティ上の注意点

`--dangerously-skip-permissions` フラグを使う場合、**そのフラグは全チームメイトに伝播します**。1人の許可が全員に適用される設計です。このフラグの使用はサンドボックス環境に限定し、本番コードベースでの使用は避けてください。

## CLAUDE.md の設計がAgent Teamsの土台になる

Agent Teamsを動かして最初に気づいたのが、**各エージェントは起動時にCLAUDE.mdを読み込む**という事実です。Team leadも各Teammateも、全員が同じCLAUDE.mdを参照しながら動作します。

ここに役割定義・禁止事項・品質基準が書かれていないと、エージェントは「自分が何をすべき役割なのか」を推測で補いながら動くことになります。

私のCLAUDE.mdには、Agent Teams向けに以下のセクションを追加しています。

```markdown
## Agent Teams 運用ルール

### チーム構成原則
- 必ずデビルズアドボケイト（DA）役を1名配置すること
- DA役は実装・提案に対して批判的レビューを行い、承認を与える役割
- DAの指摘を無視した実装は無効とする

### 各ロールの責務
- organizer: タスク分解・依存関係の整理・進行管理
- documenter: ドキュメント作成・更新・整合性確認
- security: セキュリティ観点でのコードレビュー・設定確認
- devils-advocate: 全成果物への批判的レビュー・品質ゲート

### 禁止事項
- APIキー・シークレットのハードコード
- 本番環境への直接操作
- 他エージェントの成果物を無断で変更すること
```

このような「チームの憲法」をCLAUDE.mdに持つことで、エージェント間の役割衝突が減り、DAの指摘が実装チームに適切に届くようになります。

## 実践1: 5エージェントで環境監査

### チーム構成

Mac mini M4 Pro のセットアップ作業で組んだ最初のチームです。

| エージェント | 役割 |
|---|---|
| Team lead | タスク分解・進行管理 |
| organizer | 環境整合性の確認・改善計画の立案 |
| documenter | セットアップ手順書・CLAUDE.mdの更新 |
| security | セキュリティ設定の監査・改善提案 |
| devils-advocate | 全提案・成果物への批判的レビュー |

### Team lead への指示

```
環境監査チームを立ち上げます。

【目的】
Mac mini M4 Pro の開発環境（Node.js, Python, Docker, gcloud）の整合性確認と
CLAUDE.md の初版作成。

【タスク割り当て】
- organizer: ~/dev/ 以下のディレクトリ構成確認、不整合リストアップ
- documenter: セットアップ手順書の初稿作成、CLAUDE.md の骨格作成
- security: SSH設定、.env管理、Gitフック設定のセキュリティ確認
- devils-advocate: organizer・documenter・security の成果物レビュー

【制約】
- 本番環境への変更禁止
- ファイル削除は必ず確認を取ること
- DAのレビューが完了するまで成果物をファイルに書き出さないこと
```

### 結果と各エージェントの出力

4つの並列タスクが約15分で完了しました。1エージェントで順番に実行した場合と比較して、体感で3倍程度のスループットです。

特に効果があったのは、**DA（devils-advocate）の存在**でした。

`security` エージェントが「`.gitignore` に `.env` が含まれているため問題なし」と報告した箇所に対し、DA が以下を指摘しました。

```
【DA指摘 — CRITICAL】
.gitignore で .env を除外していても、過去にステージングされた履歴がある場合は
git log --all でシークレットが参照可能です。

確認必須:
$ git log --all -- .env
$ git log --all -p | grep -E "API_KEY|SECRET|TOKEN"

既存の履歴がある場合は BFG Repo Cleaner または git filter-repo での
履歴浄化が必要です。既存コミット履歴の確認なしに「問題なし」と結論づけることは
リスクがあります。
```

この指摘により、実際に1つのリポジトリで過去のステージング履歴を発見し、対処できました。**実装エージェントだけのチームでは素通りしていた問題**です。

## 実践2: 4エージェントで並列タスク処理

### 5人→4人への変更理由

初回の5エージェント運用で気づいたのは、**Team leadが実質的にorganizerと重複していた**ことです。Team leadはタスク分配・進行管理の役割を持つため、別途organizerを立てると役割が曖昧になります。

2回目はorganizerを廃止し、Team leadにその機能を統合した4エージェント構成に変更しました。

| エージェント | 役割 |
|---|---|
| Team lead | タスク分解・依存関係管理・進行管理 |
| bot-updater | Discord Bot のコード更新・テスト |
| structure-updater | ディレクトリ構造・設定ファイルの整理 |
| cleanup-analyst | 不要ファイル・重複コードの分析・削除候補リスト作成 |
| devils-advocate | 全変更内容のレビュー（Round 2まで） |

### 依存関係のあるタスクの扱い

並列化できないタスクも存在します。たとえば `structure-updater` が完了した後に `cleanup-analyst` が動く必要がある場合、Task listに依存関係を明示します。

Team leadへの指示でこう書きます。

```
【タスク依存関係】
1. [PHASE-1] bot-updater と structure-updater は並列実行可能
2. [PHASE-2] cleanup-analyst は structure-updater の完了後に開始
3. [REVIEW] devils-advocate は PHASE-2 完了後に全成果物をレビュー
4. [ROUND-2] REVIEW で HIGH指摘が出た場合、該当エージェントが修正後に再レビュー

PHASE間の移行は Task list の [DONE] ステータスを確認してから行うこと。
```

### DAラウンド2の運用

DAの指摘は深刻度（CRITICAL / HIGH / MEDIUM / LOW）で分類し、HIGH以上は必ず修正してからDAに再レビューを依頼します。

```
【DA レビュー指摘分類】
CRITICAL: 即座に修正、リリース不可
HIGH:     同セッション内で修正必須
MEDIUM:   次回セッションで対応可
LOW:      メモとして記録、任意対応
```

このラウンド制を設けることで、「DAが指摘したが誰も直さなかった」という状況を防げます。

## Agent Teams運用のベストプラクティス

### 1. デビルズアドボケイトを必須メンバーにする

最も重要な知見です。実装系エージェントだけのチームは、**全員が「タスクを完了させる方向」に動く**ため、リスク・矛盾・見落としへの抵抗力が低くなります。

DAを1名置くだけで、以下が自然に機能します。

- セキュリティ上の見落としへの指摘
- ハルシネーションの検出（「このライブラリのAPIはそういう仕様ではない」）
- 過剰実装の抑制（「この機能は要件にないのでは？」）
- 整合性の確認（「documenterの手順とsecurityの設定が矛盾している」）

DAなしでAgent Teamsを使うのは、コードレビューなしでマージするのと同じです。

### 2. チーム人数は3〜4名が最適

5名以上になると、Team leadのコーディネーションコストが増大し、かえって非効率になるケースがあります。

また、**トークン消費は人数に比例して増加します**。5エージェントチームでは、単独エージェントの約5倍のトークンを消費します。

| 構成 | 用途 | 目安 |
|---|---|---|
| 2名（lead + specialist） | 単一ドメインの深い作業 | シンプルなレビュー等 |
| 3名（lead + 2専門家） | 中程度の並列化 | 実装 + DA |
| 4名（lead + 3専門家） | 本格的な並列開発 | 実装2名 + DA |
| 5名以上 | 大規模プロジェクト向け | コスト注意 |

### 3. タスクの粒度設計

Agent Teamsが効果を発揮するタスクの条件があります。

**向いているタスク**
- 独立して完結できる（他エージェントの出力に依存しない）
- 1〜3時間程度の作業量
- 成果物が明確（「〇〇.mdを作成する」「〇〇の関数をリファクタリングする」）

**向いていないタスク**
- 前のタスクの出力が必須の入力になる（依存度が高い）
- 5分以内で終わる軽微な修正
- 対話的な確認が多く必要な設計作業

### 4. コスト管理のポイント

Opus 4.6の料金は入力$5/Mトークン、出力$25/Mトークンです（2026年2月時点）。5エージェントチームで大規模タスクを実行すると、数百万トークンに及ぶことがあります。

コスト管理のためにやっていること。

- **タスクの事前分解**: Team leadへの指示を詳細にすることで、エージェントの試行錯誤を減らす
- **フェーズ分割**: 全タスクを一度に渡さず、フェーズごとに承認を挟む
- **Sonnetの活用**: 複雑な推論が不要なタスク（ドキュメント整形、コード整理）はSonnetを使う
- **セッション開始前の設計**: コンテキストが整っていない状態でAgentを起動しない

## 失敗談と改善ポイント

### 失敗1: CLAUDE.mdがなかった状態で起動

最初の試行では、CLAUDE.mdが不完全な状態でAgent Teamsを起動しました。各エージェントが「どこまでやっていいか」を自己判断し始め、あるエージェントが別のエージェントが担当しているファイルを編集するという競合が発生しました。

**改善策**: CLAUDE.mdに明確な「担当ドメインと編集禁止ファイル」を書いておく。

```markdown
## エージェント別ファイル担当範囲

- organizer: CLAUDE.md のみ編集可
- documenter: docs/ 以下のみ編集可
- security: .github/ および設定ファイル（*.config.js, .env.example）のみ編集可
- 他エージェントの担当範囲への変更は Team lead の承認を得ること
```

### 失敗2: セッション再開を試みた

Agent Teamsはセッション再開をサポートしていません（2026年2月時点）。途中でネットワークが切れた際に再接続を試みましたが、セッションは新規扱いとなり、途中の状態が失われました。

**改善策**: 長時間タスクはフェーズを短く分割し、各フェーズの成果物をファイルに書き出してから次フェーズへ進む設計にする。

### 失敗3: DAの指摘を後回しにした

「DAレビューは最後でいい」と考え、全実装完了後にDAを走らせたところ、CRITICAL指摘が出てほぼ全体を作り直すことになりました。

**改善策**: DAをフェーズごとに組み込む。PHASE-1完了時点でDAレビューし、CRITICAL/HIGHをつぶしてからPHASE-2へ進む。

## まとめ

Agent Teamsを2回実践して得た結論をまとめます。

**「並列化」ではなく「チーム設計」が本質**

Agent Teamsの価値は、タスクを速く終わらせることだけではありません。役割を分けることで「実装する視点」「批判する視点」「文書化する視点」を同時に持てることが、単独エージェントにはない強みです。

特に、デビルズアドボケイトを常設メンバーにする運用は、個人開発者や小規模チームにとって実用的な品質保証の仕組みになります。コードレビュアーが常にいる状態を、AIで実現できるからです。

**CLAUDE.mdへの投資が先**

CLAUDE.mdが整備されていない状態でAgent Teamsを使っても、エージェントがバラバラに動くだけです。チームのルール・役割・禁止事項を先に書いておくことが、Agent Teamsの前提条件です。

今後は、Subagentsとのハイブリッド構成（Agent TeamsがサブエージェントをさらにSpawnする設計）も試してみる予定です。
