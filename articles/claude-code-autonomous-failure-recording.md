Claude Codeを本格的に業務運用すると、ある壁にぶつかります。「AIが同じミスを繰り返す」という問題です。

推測を事実として報告する。指示範囲外の変更を勝手に行う。検証せずに「完了しました」と報告する。セッションが変わるたびにコンテキストがリセットされ、同種のミスが何度も再発する。人間の開発者であれば過去の失敗から学んで行動を変えますが、AIエージェントにはその仕組みがありません。

本記事では、Claude Codeに「自律的再発防止記録システム」を実装し、AIエージェントが自分のミスを検知・記録・学習するフィードバックループを構築する方法を解説します。約3か月間の実運用で20件超のケースを蓄積し、再発率を大幅に低減できた手法の全容をお伝えする内容です。

## 全体アーキテクチャ: 3層構造

このシステムは3つの層で構成されています。

```
┌──────────────────────────────────────────────────┐
│  Layer 1: Knowledge File（プロトコル定義）          │
│  → failure-recording-protocol.md                  │
│  → 3種トリガー / 5ステップ / エスカレーション基準   │
├──────────────────────────────────────────────────┤
│  Layer 2: Hooks（自動検証）                        │
│  → session-reminder.sh（Stop hook）               │
│  → session-guard.sh（PreToolUse hook）            │
│  → settings.json でイベント定義                    │
├──────────────────────────────────────────────────┤
│  Layer 3: 組織記憶（パターン蓄積・昇格）            │
│  → failure-cases.md（全Case記録）                  │
│  → MEMORY.md（MUST項目への昇格）                   │
│  → フィードバックメモリ（個別の学習記録）           │
└──────────────────────────────────────────────────┘
```

Layer 1がAIの行動規範を定義し、Layer 2が記録漏れを機械的に検証し、Layer 3が組織としての記憶を蓄積・強化していく構造です。

## Layer 1: プロトコル定義

### 3種のトリガー設計

ミスの検知は3つの経路で行います。

```markdown
## トリガーA: ユーザー起因の検知
- ユーザーから修正指摘を受けた（「違う」「そうじゃない」「なぜこうした？」）
- ユーザーが同じ指示を2回以上繰り返した
- ユーザーの口調が変化した（丁寧語→端的な指摘 = 不満のシグナル）

## トリガーB: 自己検知
- 自分で実装ミスに気づいた（テスト失敗、ビルドエラー等）
- デバッグの3回リトライでも解決せず停止に至った
- 前回と同じ種類のミスを繰り返していると気づいた
- 推測で報告した内容が事実と異なっていた

## トリガーC: システム検知
- hookがブロックした（かつ原因が自分のミス）
- CI/CDが失敗した（自分の変更起因）
- デプロイ後に本番で問題が発生した
```

ここで重要なのは、トリガーBの「自己検知」を明文化している点です。「記録しろ」と言われてから記録するのではなく、ミスを **検知した瞬間に自律的に記録を開始する** ことをAIの行動規範として定義しています。

### 5ステップの実行手順

トリガーが発動したら、以下の5ステップを順に実行します。

**Step 1: 停止と認識**

現在の作業を中断し、「ミスが発生しました」とユーザーに即座に報告します。何が壊れたか、誰に影響するか、データ損失はあるかを明示する段階です。隠蔽や矮小化は厳禁というルールを設けています。

**Step 2: 構造化記録**

セッション記録の `Errors & Failures` セクションに、以下のテンプレートで記録します。

```markdown
### [HH:MM] {エラー名/ミスの概要}
- **トリガー**: {A/B/Cのどれで検知したか}
- **症状**: {何が起きたか — 事実のみ}
- **暫定原因**: {発覚時点の推測}
- **確定原因**: {検証後に更新}
- **影響**: {ユーザー/本番/データへの実害}
- **対処**: {何をして解決したか}
- **所要時間**: {発覚→解決の時間}
- **既知パターン照合**: {failure-cases.md の Case N に該当 / 新規パターン}
```

「暫定原因」と「確定原因」を分離しているのがポイントです。AIは原因を即座に断定しがちですが、最初の推測が間違っていることは珍しくありません。2段構成にすることで、推測と検証済みの事実を区別する習慣をAIに植え付けています。

**Step 3: 既存パターンとの照合**

`failure-cases.md` に蓄積された過去のケースと照合します。既知パターンの再発であれば、なぜ既存の防止策で防げなかったかを分析し、防止策を強化します。新規パターンであれば Step 4 へ進みます。

**Step 4: 新規パターンの記録**

新規パターンの場合、`failure-cases.md` に以下の構造で追記します。

```markdown
## Case N: {ミスの概要}（{日付}）

**何が起きたか**: {客観的な事実}

**根本原因**:
1. {直接原因}
2. {構造的原因（なぜその直接原因が発生しうる状態だったか）}
3. {判断の誤り（なぜその構造的問題を見逃したか）}

**再発防止策**:
### A. {技術的対策}
- {具体的な手順やチェック項目}

### B. {判断レベルの対策}
- {どういう状況で何を確認するか}
```

根本原因を3層で記録するのがこのフォーマットの核心です。「タイポだった」で終わらせず、「なぜタイポが本番に到達したか」「なぜレビューで見逃したか」まで掘り下げます。

**Step 5: エスカレーション判定**

ミスの深刻度と再発回数に応じて、記録の保存先をエスカレーションします。

| 条件 | エスカレーション先 |
|------|------------------|
| 初回発生 | セッション記録 + failure-cases.md |
| 同カテゴリ2回目 | + MEMORY.md に MUST 項目として追加 |
| 同カテゴリ3回目 | + Knowledge File化 or hookによる物理ブロック |
| 信頼毀損（虚偽報告等） | 即座に MEMORY.md MUST + フィードバックメモリ |

「同カテゴリ」の判定基準は「同じ防止策で防げたはずのミス」と定義しています。表面的には異なるミスでも、防止策が共通なら同カテゴリとして扱い、エスカレーションの対象にする考え方です。

## Layer 2: hookによる自動検証

プロトコルを定義しただけでは、AIが記録を忘れる可能性があります。Claude Codeの hooks 機能を使って、記録漏れを機械的に検証する仕組みを実装しました。

### settings.json でのhook定義

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/session-guard.sh",
            "timeout": 10
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/session-reminder.sh",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

2種類のhookを組み合わせています。

- `PreToolUse`（Write/Edit時）: セッション記録ファイルへの書き込みが、正しいセッションのファイルに対するものかをアンカーファイルで検証
- `Stop`（セッション終了時）: セッション中にエラーの痕跡があるのに `Errors & Failures` セクションが空なら警告

### Stop hookの実装: 記録漏れ検証

セッション終了時に動作する `session-reminder.sh` の核心部分を見てみましょう。

```bash
# ===== エラー記録漏れ検証 =====
# セッション記録内の Actions セクションからエラー/ミスの痕跡を検索
ACTIONS_CONTENT=$(sed -n '/^## Actions/,/^## Outcomes/p' "$LATEST" 2>/dev/null)
ERROR_SIGNALS=$(echo "$ACTIONS_CONTENT" | grep -ciE \
  '(修正指摘|ユーザー指摘|やり直し|再実行|revert|rollback|「違う」|「そうじゃない」|なぜこうした|誤って|誤設定|誤判断|虚偽|不具合が発生|障害発生)' \
  2>/dev/null || echo "0")

# Errors & Failures セクションの内容を確認（テンプレートのみは除外）
EF_CONTENT=$(sed -n '/^## Errors & Failures/,/^## /p' "$LATEST" 2>/dev/null \
  | grep -v '^## ' \
  | grep -v '^\[エラー名\]' \
  | grep -v '^\- \*\*症状\*\*:$' \
  | grep -v '^$' \
  | head -3)

if [ "$ERROR_SIGNALS" -gt 0 ] && [ -z "$EF_CONTENT" ]; then
  echo "FAILURE RECORDING ALERT: セッション中にエラー/ミスの痕跡が${ERROR_SIGNALS}件ありますが、"
  echo "   Errors & Failures セクションが未記入です。"
  echo "   → failure-recording-protocol.md に従い、原因と再発防止策を記録してください。"
fi
```

このhookの設計で注意したポイントが2つあります。

1つ目は **false-positive（過検知）の防止** です。grepパターンに「エラーを修正した」「問題を解決した」のような正常完了の表現を含めると、すべてのセッションで誤警告が出てしまいます。パターンを「ユーザー指摘」「やり直し」「revert」など、明確に問題発生を示す表現に限定しました。

2つ目は **アンカー解決時のみ検証を実行する** 点です。セッションアンカー（どのセッション記録ファイルが現在のセッションに紐づくか）が特定できない場合、別セッションのファイルを誤検知するリスクがあるため、検証をスキップします。

### PreToolUse hookの実装: セッション混在防止

`session-guard.sh` は、別セッションの記録ファイルへの書き込みを物理的にブロックします。

```bash
# Claude Code PID をキーとしたアンカーファイルで「この会話のセッション」を記録
MY_ID=$("$HOME/.claude/hooks/lib/get-session-id.sh")
MY_ANCHOR="$ANCHOR_DIR/${MY_ID}.anchor"

# このセッションのアンカーが存在する場合
if [ -f "$MY_ANCHOR" ]; then
  EXPECTED=$(cat "$MY_ANCHOR" 2>/dev/null)
  if [ "$FILE_PATH" = "$EXPECTED" ]; then
    # 正しいファイル → 許可
    echo '{"hookSpecificOutput":{"permissionDecision":"allow"}}'
  else
    # 別ファイル → ブロック
    block_json "セッション混在防止: このセッションの記録ファイルは ${EXPECTED_NAME} です。"
  fi
fi
```

PIDベースのアンカーファイルで「このClaude Codeセッション = このセッション記録ファイル」という1対1対応を保証し、別プロジェクトの記録への誤書き込みを防ぎます。staleなアンカー（プロセス終了済み、24時間超過、PID再利用）の自動クリーンアップも組み込んであるため、メンテナンスは不要です。

## Layer 3: 組織記憶の育て方

### failure-cases.mdが「生きた文書」になるまで

`failure-cases.md` は単なるエラーログではありません。組織（この場合は人間 + AIエージェント）が過去のミスから学び続けるための **組織記憶** として機能します。

実運用で蓄積されたケースを、カテゴリ別に分類すると以下のパターンが見えてきました（2026年4月時点の代表的な分類）。

| カテゴリ | 件数 | 代表例 |
|---------|-----|------|
| スコープ逸脱 | 2件 | 指示外のコンテンツ追加、勝手な設計変更 |
| 検証不足 | 5件 | リダイレクト先未確認、ビルド成功=完了と報告 |
| 記録混在 | 2件 | 別プロジェクトのログを同じファイルに記録 |
| 推測の事実化 | 3件 | 未検証の推測を断定として報告 |
| 情報源の無視 | 2件 | ユーザー提供資料を参照せずに推測 |
| デザイン空回り | 1件 | 目視なしでCSS修正を繰り返す |
| コンテキスト盲信 | 1件 | 要約を検証せず行動 |

「検証不足」が最多であり、これがAIエージェント運用における最大のリスクカテゴリであることがわかります。

### 実際のCase: 推測を事実として報告した例

最も教訓的だったケースを1つ紹介します（匿名化済み）。

あるWebサイトの記事公開作業で、11本の記事の `publishedAt` を翌日以降に分散させ、「1日1本で公開されます」とユーザーに報告しました。しかし実際には、コードベースのコンテンツ取得関数に未来日付のフィルタリング機能が実装されておらず、pushした瞬間に全11本が即時公開されていたのです。

根本原因の3層分析はこうなります。

1. **直接原因**: `publishedAt` を未来日付にすれば自動的にスケジュール公開になると思い込んだ
2. **構造的原因**: frontmatterのメタデータと表示制御ロジックの区別をしていなかった
3. **判断の誤り**: 「1日1本で公開されます」という報告が、実際の検証に基づかない推測だった

このケースから導いた再発防止策は「公開制御を伴う操作は必ずコードレベルで検証する」「推測と事実を区別して報告する」という2点です。具体的には、pushする前に `grep -n "publishedAt\|draft\|filter.*date" src/lib/*.ts` でフィルタリングロジックの存在を確認するステップを追加しました。

### もう1つのCase: 本番APIへの推測投入

メールアドレスを確認せずに推測で本番APIに投入してしまったケースも印象的です。設定ファイルに「メールドメイン: example.com」と書いてあることから、`user@example.com` が存在するだろうと推測し、検証なしでニュースレター登録APIをコールしました。

実際にはそのメールアドレスは存在せず、外部メール送信サービスがbounce処理を実行。本番データベースにゴミデータが作成されるという事態になりました。

このケースが教えてくれたのは「ドメインを知っている ≠ アドレスを知っている」という原則と、「お任せします」という指示は細部の推測を承認するものではないという判断基準です。

### エスカレーションの実例

同カテゴリのミスが繰り返されると、防止策の強度が段階的に上がっていきます。

セッション記録の混在問題を例にとると、初回発生時は `failure-cases.md` に記録するだけでした。しかし2回目が発生した際、hookが検知できない「アンカー済みファイルへの別PJ追記」というパターンだと判明。これを受けて、`MEMORY.md` にMUST項目として「pre-commit hook応答時にアンカー先PJ名と現作業PJの一致確認必須」を追加しました。

```
MUST 0.5: セッション記録の完全分離
- 1PJ = 1記録。混在禁止。PJ変更→新セッション作成
- フックが防げないパターン: アンカー済みファイルへの別PJ追記。判断レベルで防ぐ
- pre-commit hook対応時: アンカー先PJ名と現作業PJの一致確認必須
```

hookで物理的にブロックできない問題を、MEMORY.md（AIが毎セッション自動読み込みする記憶ファイル）で判断レベルで防ぐという多層防御のアプローチです。

## 記録の品質基準: 「気をつける」を禁止する

再発防止策の品質は、このシステム全体の効果を左右します。以下の基準を設けています。

**良い記録の4条件:**

1. 原因が3層（直接原因 → 構造的原因 → 判断の誤り）
2. 再発防止策が具体的（「{状況}で{アクション}を実行する」の形式）
3. 検証可能（防止策が守られているかを第三者が確認できる）
4. 適用範囲が明確（いつ・どの作業で・何をチェックするか）

**禁止される表現:**

- 「次回は注意する」→ 何に注意するのか不明
- 「確認を徹底する」→ 何を・いつ・どう確認するのか不明
- 「ユーザーに確認する」→ どの状況で確認が必要なのか不明

この基準により、`failure-cases.md` の各Caseは「特定の状況で特定のアクションを実行する」形式の、機械的に適用可能な再発防止策に収束していきます。

## 「毎回答の最終チェック」統合リスト

各Caseから抽出したチェック項目を、2つのカテゴリに分けて運用しています。

**常時スキャン（全回答前に確認）:**

```
1. 指示されていない内容を追加していないか？
2. セッション記録のPJ名と現作業PJは一致しているか？
3. ユーザーへの報告に未検証の推測が含まれていないか？
```

**特定作業時のみ確認:**

```
- 公開・デプロイ時 → 公開制御ロジックをコードで確認したか？
- データ抽出時 → grepではなくランタイム（TypeScript実行等）を使ったか？
- デザイン変更時 → スクリーンショットで目視確認したか？
- 「完了」報告時 → 本番環境での動作を検証したか？
```

これらのチェックリストは `failure-cases.md` の各Caseと紐づいており、ケースが追加されるたびに自動的にチェック項目も増えていく仕組みです。

## debugging-protocol との役割分担

技術的なデバッグ手順（`debugging-protocol.md`）と再発防止記録（`failure-recording-protocol.md`）は、明確に役割を分離しています。

```
debugging-protocol.md:
  → エラーの技術的解決手順（リトライ、原因特定、修正）
  → 発動: エラー発生時
  → 記録先: 05_errors/error-*.md

failure-recording-protocol.md:
  → ミスの記録と再発防止（原因分析、パターン蓄積、エスカレーション）
  → 発動: debugging-protocol の3回リトライ失敗後
  → 記録先: セッション記録 + failure-cases.md
```

デバッグプロトコルが「目の前の火を消す」役割なら、再発防止プロトコルは「次の火事を防ぐ」役割です。両方を同時に参照しながら、技術的解決と記録・再発防止を並行実行します。

## 導入手順: あなたの環境で再現する

### Step 1: ファイル構成

以下の3ファイルを作成します。

```
~/.claude/
├── knowledge/
│   └── failure-recording-protocol.md   # Layer 1: プロトコル定義
├── hooks/
│   └── session-reminder.sh             # Layer 2: 自動検証
└── projects/
    └── {your-project}/
        └── memory/
            └── failure-cases.md         # Layer 3: 組織記憶
```

### Step 2: CLAUDE.md にトリガーを登録

```markdown
## MUST Rules
10. **ミス発覚・ユーザー指摘・同じ指示の繰り返し検知時は自律的に再発防止記録**:
    即座に原因・防止策を記録。詳細は `failure-recording-protocol.md`
```

Knowledge Filesのルーティングテーブルにも登録します。

```markdown
| ミス発覚・ユーザー指摘時 | `failure-recording-protocol.md` | 自律記録トリガー、5ステップ、エスカレーション基準 |
```

### Step 3: settings.json にhookを登録

以下はStop hookの最小構成です。実運用では、Layer 2で紹介した `session-guard.sh`（PreToolUse hook）も追加することで、記録漏れの自動検証だけでなくセッション混在の物理ブロックも機能します。

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/session-reminder.sh",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

### Step 4: failure-cases.md の初期化

```markdown
# Failure Cases（再発防止用）

（ケースが発生するたびにここに追記されていく）
```

最初は空で問題ありません。運用を始めれば、ケースは自然に蓄積されていきます。

### Step 5: MEMORY.md にプロトコルを登録

```markdown
## MUST: 自律的再発防止記録（Autonomous Failure Recording）
- ミス検知時に即座に記録を開始する（ユーザー指摘を待たない）
- 3種トリガー: (A)ユーザー修正指摘 (B)自己検知 (C)hook/CIブロック
- 5ステップ: 停止+報告 → セッション記録 → パターン照合 → 新規Case → エスカレーション
```

## 3か月運用して見えてきたこと

### 定量的な効果

20件超のケースを蓄積した結果、明確に減少したミスのカテゴリがあります。「セッション記録の混在」はhookによる物理ブロック導入後、完全に再発がなくなりました。「スコープ逸脱」も、毎回答のチェックリストに追加してからは発生していません。

一方、「検証不足」は形を変えて再発する傾向があります。これは、検証すべき対象が作業によって変わるため、チェックリストの網羅性に限界があるためでしょう。ただし、同じ対象での検証不足は確実に減少しています。

### 設計上の重要な判断

**hookのfalse-positive防止が最重要**: grepパターン設計を間違えると、正常なセッションでも毎回警告が出て、ユーザーが警告を無視するようになります。「エラーを修正した」を検知しないよう、パターンを「問題発生を示す表現」に厳密に限定しました。

**「同カテゴリ」の定義が運用の鍵**: 表面的には異なるミスでも「同じ防止策で防げたはずのミス」を同カテゴリとすることで、エスカレーション判定が実用的になりました。例えば、アフィリエイトリンクの誤設定と引用URLの不一致は、どちらも「登録後のリンク先検証」で防げたミスなので同カテゴリ「検証不足」として扱います。

**hookで防げない問題は判断レベルで補完する**: すべての問題を自動検証で防ぐことは不可能です。hookで物理的にブロックできる問題と、MEMORY.mdで判断レベルで防ぐ問題を分けて考えることが実運用では不可欠でした。

## まとめ

AIエージェントの品質改善は、プロンプトの改善だけでは限界があります。セッションを跨いで学習が蓄積される仕組み、つまり「組織記憶」が必要です。

本記事で紹介したシステムの要点は3つです。

1. **プロトコルでAIの行動規範を定義する**: 3種のトリガーと5ステップの手順を明文化し、AIが自律的にミスを検知・記録する
2. **hookで記録漏れを機械的に検証する**: 人間が指摘しなくても、システムが記録の欠落を検出する
3. **エスカレーションでルールを段階的に強化する**: 同じカテゴリのミスが繰り返されるたびに、防止策の強度を上げていく

「AIが間違えること」自体は避けられません。しかし「同じ間違いを繰り返すこと」は、仕組みで防げます。failure-cases.md に蓄積されたケースが増えるほど、AIエージェントの判断精度は向上していく。このフィードバックループこそが、AIエージェントの「自己修復」能力の本質です。
