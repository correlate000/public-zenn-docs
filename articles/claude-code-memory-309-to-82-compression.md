## はじめに

Claude Codeを業務利用して半年以上が経った頃、ある限界に直面しました。

```
> WARNING: MEMORY.md is 309 lines (limit: 200).
> Only part of it was loaded. Keep index entries to one line
> under ~200 chars; move detail into topic files.
```

Claude Code の Auto Memory（MEMORY.md）が膨張し、200行の推奨上限を大幅に超過。セッション開始時に **一部しか読み込まれない** 状態に陥っていたのです。

このとき検討したのが2つの選択肢。AnthropicがClaude Code v1.0.33で導入した **AutoDream** （自動メモリ整理機能）に任せるか、手動で圧縮するか。結論としてAutoDreamの導入は見送り、手動で309行を82行まで圧縮しました。73%の削減です。

本記事では、その具体的な手法と判断基準を共有します。

:::message
本記事は「CLAUDE.md設計ガイド」シリーズの第4弾です。第1弾（[CLAUDE.md設計ガイド](https://zenn.dev/correlate_dev/articles/claude-md-guide)）、第2弾（[Knowledge Files分離](https://zenn.dev/correlate_dev/articles/claude-code-knowledge-files)）、第3弾（[メモリ設計](https://zenn.dev/correlate_dev/articles/claude-code-memory-design)）を前提としています。
:::

## なぜAutoDreamを見送ったのか

AutoDreamは `claude config set autoMemory dream` で有効化できるClaude Codeの自動メモリ整理機能です。セッション終了時にAIがメモリを自動で整理・圧縮してくれるため、手動管理の手間を省ける...はずでした。

導入を見送った理由は3つあります。

### 1. メモリ保存先のカスタマイズとの相性問題

2026年4月時点で、AutoDreamのメモリ保存先に関する挙動が安定しない報告がコミュニティで散見されていました。メモリの保存先ディレクトリをカスタマイズしている環境では、意図しない場所にファイルが生成される可能性がある。私の環境ではメモリファイルの配置に明確な設計思想があったため、この不確実性は許容できませんでした。

### 2. frontmatter保持の保証がない

私のトピックファイルには以下のようなfrontmatterが設定されています。

```yaml
---
name: 嘘をつかない・事実と異なる報告を絶対にしない
description: 作業を止めたのに「止めていません」と言う等の虚偽報告を絶対にしない
type: feedback
---
```

AutoDreamがこのfrontmatterを保持してくれる保証がないため、メタデータが消失するリスクがありました。

### 3. 圧縮ロジックの不透明さ

どのエントリが要約され、どの情報が削除されるのか。AutoDreamの判断基準が明確でないため、業務クリティカルなルール（例：「虚偽報告の絶対禁止」「PRマージ先はstagingが原則」）が誤って簡略化される可能性を排除できません。

手動管理であれば「何を残し、何を分離し、何を削除したか」を完全に制御可能。情報の重要度は自分が一番よく知っているという判断から、手動圧縮を選択しました。

## 圧縮前のMEMORY.mdの問題

まず、肥大化したMEMORY.mdの構造的問題を整理します。

### 問題1: インライン詳細の肥大化

圧縮前のMEMORY.mdには、1エントリに5行以上の詳細説明がインラインで書かれているものが多数ありました。

```markdown
## 🚨 MUST: 自律的再発防止記録（Autonomous Failure Recording）
- ミス検知時に即座に記録を開始する（ユーザー指摘を待たない）
- **3種トリガー**: (A)ユーザー修正指摘・同じ指示2回以上
  (B)自己検知（debugging-protocol 3回リトライ後停止・同種ミス再発）
  (C)hook/CIブロック
- **5ステップ**: ①停止+報告 → ②セッション記録 → ③failure-cases.md既存パターン照合
  → ④新規Case作成 → ⑤エスカレーション判定
- **エスカレーション**: 初回→failure-cases.md / 同カテゴリ2回目→feedbackメモリ化
  / 3回目→Knowledge File or hook物理ブロック / 信頼毀損・不可逆→即MUST
- **「同カテゴリ」= 同じ防止策で防げたはずのミス**
- session-reminder.sh（Stop hook）が記録漏れを自動検証
- debugging-protocol.md は技術的解決、本プロトコルは記録・再発防止
- 詳細: ~/.claude/knowledge/failure-recording-protocol.md
```

この1エントリだけで8行を消費しています。

### 問題2: 同種情報の散在

見積もり関連のルールだけでも複数箇所に分散し、合計20行以上を占めていたケースも。

### 問題3: 200行上限の仕様

Claude CodeはMEMORY.mdが200行を超えると、 **一部しか読み込まない** 仕様になっています。つまり309行のMEMORY.mdでは、後半に書かれたルールがセッション開始時に読み込まれず、無視される危険がありました。

## 圧縮の具体的手法

圧縮のアプローチは **「MEMORY.mdを参照インデックスに徹させる」** という一点に集約されます。

### 手法1: インライン詳細をトピックファイルに分離

10行以上のインライン詳細を持つエントリを、個別のトピックファイルに切り出しました。

**Before（MEMORY.md内にインライン記述）:**

```markdown
## 🚨 MUST: 自律的再発防止記録（Autonomous Failure Recording）
- ミス検知時に即座に記録を開始する（ユーザー指摘を待たない）
- **3種トリガー**: (A)ユーザー修正指摘... (B)自己検知... (C)hook/CIブロック
- **5ステップ**: ①停止+報告 → ②セッション記録 → ③照合 → ④Case作成 → ⑤判定
- **エスカレーション**: 初回→failure-cases / 2回目→feedback / 3回目→Knowledge File
（計8行）
```

**After（MEMORY.mdは1行のインデックス + 別ファイルに詳細）:**

```markdown
## 🚨 MUST: 自律的再発防止記録 — 詳細: failure-recording-protocol.md
```

詳細は `~/.claude/knowledge/failure-recording-protocol.md` に格納済み。MEMORY.mdには「何のルールか」と「どこを見ればよいか」だけを残す方針です。

### 手法2: 1エントリ = 1行ルール

圧縮後のMEMORY.mdでは、原則として **1つのエントリを1行（200文字以内）で記述** しています。

```markdown
## MUST: pre-commit hook変更時はdry-run確認必須（feedback_precommit_dryrun.md）

## MUST: 丁寧語統一 — 全出力で「です/ます」。タメ口禁止

## MUST: Vercelデプロイ先は correlate-pro — --scope correlate-pro 必須
```

このルールの効果は明確で、各エントリが何を言っているのかを **見出しだけで把握できる** ようになりました。Claudeがメモリを参照する際も、1行でルールの要旨を理解し、詳細が必要なときだけトピックファイルを読みに行く、という動線が成立します。

### 手法3: メモリの種類による分類と命名規則

トピックファイルには4種類のプレフィックス命名規則を設けました。

| プレフィックス | 用途 | 例 | 現在のファイル数 |
|-------------|------|---|-------------|
| `feedback_` | ユーザーからの指摘・修正フィードバック | `feedback_no_lies.md` | 83個 |
| `project_` | プロジェクト固有の状態・戦略 | `project_gikaiscope.md` | 22個 |
| `reference_` | 環境・手順のリファレンス | `reference_claude_dispatch.md` | 6個 |
| `user_` | ユーザー個人情報 | `user_orcid.md` | 1個 |

最も多いのが `feedback_` ファイル（83個）です。これはClaude Codeの運用で発生したミスや改善点を1件ずつ記録したもの。同じミスを二度と繰り返さないための学習記録として機能しています。

各ファイルの典型的な構造は以下の通りです。

```markdown
---
name: 嘘をつかない・事実と異なる報告を絶対にしない
description: 虚偽報告を絶対にしない。事実をそのまま報告する。
type: feedback
---

ユーザーから「なんで途中で止めたのに止めてません、って
平気で嘘つくの？」と強い指摘（2026-03-22）。

**事例:**
1. 作業を途中で止めてユーザーに確認を求めたのに、
   「止めていません」と返答
2. スクショ確認せずに「きちんと配置されました」と報告

**How to apply:**
1. 止めたなら「止めました」と言う
2. 確認していないなら「未確認です」と言う
3. 間違えたなら「間違えました」と言う
```

frontmatterで `type` を明示しているため、機械的な集計やフィルタリングにも対応可能。ファイル名だけで種別が判別できるのも運用上の大きなメリットです。

### 手法4: Knowledge Filesとの役割分担

MEMORY.mdのトピックファイル以外に、 **Knowledge Files** （`~/.claude/knowledge/` 配下）という別の知識レイヤーが存在します。

両者の使い分けは明確です。

| 項目 | Knowledge Files | MEMORY.md トピックファイル |
|------|----------------|------------------------|
| 配置場所 | `~/.claude/knowledge/` | `~/.claude/projects/.../memory/` |
| 読込タイミング | CLAUDE.mdのルーティングテーブルで指定 | セッション開始時に自動読込 |
| 内容の性質 | 技術的パターン・手順書 | 行動ルール・フィードバック・プロジェクト状態 |
| 更新頻度 | 低（安定した知識） | 高（日常的に追加） |
| ファイル数 | 31個（計4,261行） | 119個（MEMORY.md含む） |

たとえば「BigQueryのMERGE文の書き方」はKnowledge Filesに、「BigQueryクエリ実行前に必ずDRY RUNする」というルールはMEMORY.mdトピックファイルに格納する、という棲み分けになります。

## 圧縮作業の実際の流れ

では、309行のMEMORY.mdを82行にした具体的なステップを紹介します。

### Step 1: 全エントリの棚卸し

まず全エントリを以下の4カテゴリに分類しました。

1. **残す（1行化）**: ルール名とリンクだけで意味が通じるもの
2. **分離（トピックファイル作成）**: 詳細説明が必要だがMEMORY.mdに全文を置く必要がないもの
3. **統合（Knowledge Filesに追記）**: 既存のKnowledge Filesに含めるべき技術的パターン
4. **削除**: 既に別の場所に記述済みの重複エントリ

### Step 2: トピックファイルへの分離

インライン詳細が5行以上あるエントリを洗い出し、個別ファイルに切り出しました。このとき重要なのが **情報欠損ゼロの原則** です。

切り出し作業では以下を必ず確認します。

```bash
# 分離前のMEMORY.mdの該当セクション行数
grep -c "MUST:" MEMORY.md  # => 58エントリ

# 分離後のトピックファイル数
ls feedback_*.md | wc -l   # => 83個

# MEMORY.mdの各エントリに対応するリンクが存在するか
grep -o '\[.*\.md\]' MEMORY.md | wc -l  # リンク数を確認
```

分離元のインライン記述と分離先のファイル内容を突合し、情報が欠落していないことを検証します。

### Step 3: MEMORY.mdのインデックス化

分離完了後、MEMORY.mdの各エントリを1行に圧縮。フォーマットは統一しました。

```markdown
## [重要度]: [ルール名] — [補足]（[リンク先ファイル名]）
```

重要度は3段階です。

- `🚨 MUST`: 違反するとユーザーの信頼を毀損する最重要ルール
- `MUST`: 遵守必須の標準ルール  
- `Topic`: プロジェクト情報や戦略メモ（違反の概念がないもの）

### Step 4: 整合性の最終検証

圧縮後に以下を確認しました。

- MEMORY.mdの全リンクが実在するファイルを指しているか
- トピックファイルからMEMORY.mdへの逆参照が整合しているか
- Claude Codeの新規セッションで「WARNING: MEMORY.md is N lines」が出ないか

## 圧縮の効果

数値で示します。

| 指標 | Before | After | 変化 |
|------|--------|-------|------|
| MEMORY.md行数 | 309行 | 82行 | 73%削減 |
| 200行上限警告 | 表示あり | 表示なし | 解消 |
| メモリ読込率 | 一部欠落 | 全件読込 | 100%に回復 |
| トピックファイル数 | 少数 | 119個 | 体系的に分離 |
| Knowledge Files | 29個 | 31個 | 2個新規作成 |
| 情報欠損 | - | 0件 | 欠損ゼロ達成 |

最も重要な効果は **メモリ読込率の100%回復** です。309行時代は後半のエントリが読み込まれず、結果としてルールが無視されることがありました。82行への圧縮で、全エントリが確実にセッション開始時に読み込まれる状態を取り戻した形です。

## 運用で効いているパターン

圧縮後の運用で特に効果を実感しているパターンを紹介します。

### パターン1: エスカレーション駆動のメモリ追加

新しいルールをMEMORY.mdに追加する基準を明確にしています。

```
初回ミス → failure-cases.md に記録（MEMORY.mdには追加しない）
     ↓
同カテゴリ2回目 → feedback_ トピックファイル作成 + MEMORY.mdに1行追加
     ↓
同カテゴリ3回目 → Knowledge File化 or pre-commit hookで物理ブロック
```

このエスカレーション構造により、MEMORY.mdが「本当に重要なルール」だけで構成される状態を維持できます。1回限りの失敗は `failure-cases.md` に留め、繰り返されたものだけがMEMORY.mdに昇格する仕組みです。

### パターン2: MEMORY.mdの定期棚卸し

月1回、以下のチェックを実施しています。

```bash
# 現在の行数確認
wc -l MEMORY.md

# 200行上限に近づいていないか
# 150行を超えたら圧縮を検討

# 最近1ヶ月で更新されていないトピックファイルの特定（macOS対応）
find . -name "*.md" -type f -mtime +30
```

MEMORY.mdが150行を超えたら圧縮を検討するという閾値を設けることで、再び309行まで膨張する事態を防いでいます。

### パターン3: frontmatter付きトピックファイル

トピックファイルにfrontmatterを付けることで、プログラマティックな操作が可能になります。

```bash
# 全feedbackファイルのname一覧を取得
grep -h "^name:" feedback_*.md | sed 's/name: //'

# type別の集計
grep -l "type: feedback" *.md | wc -l  # => 83
grep -l "type: project" *.md | wc -l   # => 22
```

将来的には、このメタデータを使って「未解決のfeedbackだけをダッシュボード表示する」「特定プロジェクトのメモリだけを抽出する」といった自動化も視野に入ります。

## AutoDreamとの比較まとめ

最後に、AutoDreamと手動管理の比較を整理しておきます。

| 観点 | AutoDream | 手動管理 |
|------|-----------|---------|
| 初期コスト | ゼロ（設定1行） | 高（設計+移行作業） |
| 運用コスト | ゼロ（自動） | 月1回の棚卸し |
| 情報の制御性 | 低（AI判断） | 高（完全に人間が制御） |
| 構造の一貫性 | 保証なし | 命名規則+frontmatterで保証 |
| 安定性リスク | あり（保存先の挙動等） | なし |
| 適する規模 | メモリ50行以下の個人利用 | メモリ100行超の業務利用 |

AutoDreamが悪いツールだというわけではありません。メモリが少量で、構造設計にこだわらない使い方であれば十分に機能するはずです。ただし、業務で100行を超えるMEMORY.mdを運用している場合、「何を残し、何を分離するか」の判断を人間が制御したい場面は多いのではないでしょうか。

## すぐ始められる3ステップ

記事を読んで「自分のMEMORY.mdも整理したい」と思った方向けに、最小限の手順をまとめます。

### Step 1: 現状把握

```bash
# MEMORY.mdの行数を確認
wc -l ~/.claude/projects/*/memory/MEMORY.md

# 200行を超えていたら圧縮対象
```

### Step 2: 5行以上のエントリを分離

MEMORY.md内で5行以上を使っているエントリを見つけ、トピックファイルに切り出します。

```bash
# memoryディレクトリに移動
cd ~/.claude/projects/<your-project>/memory/

# トピックファイルを作成
cat > feedback_example_rule.md << 'EOF'
---
name: ルール名
description: 1行で説明
type: feedback
---

詳細な説明をここに書く。
事例、適用方法、背景など。
EOF
```

MEMORY.mdには1行のインデックスだけを残します。

```markdown
## MUST: ルール名（feedback_example_rule.md）
```

### Step 3: 150行ルールを設定

MEMORY.mdが150行を超えたら棚卸しを行う、というルール自体をMEMORY.mdに書いておくのがおすすめです。

```markdown
## MUST: MEMORY.mdが150行超で棚卸し実施 — 1エントリ=1行を維持
```

## おわりに

Claude Codeの MEMORY.md は「AIの長期記憶」として強力な機能ですが、放置すると際限なく膨張します。309行に達した時点で「読み込まれないメモリ」という本末転倒な状態に陥っていました。

手動圧縮のコアは単純で、 **MEMORY.mdをインデックスに徹させ、詳細はトピックファイルに分離する** こと。これだけで73%の圧縮を達成し、全ルールが確実に読み込まれる状態を取り戻せました。

AutoDreamの進化を待つのも一つの選択肢です。しかし今日のMEMORY.mdが200行を超えているなら、手動圧縮は今すぐ効果が出る実践的な解決策と言えます。
