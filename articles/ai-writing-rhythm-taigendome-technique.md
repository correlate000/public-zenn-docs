## はじめに

AI生成の記事を読んでいると、なんとなく機械っぽさを感じることがあります。内容は正確なのに、読んでいて疲れる。同じようなリズムが延々と続く感覚。

その正体は「 ** 文末同質化 ** 」です。

`〜します。〜です。〜ます。〜します。〜です。`

こういったパターンが数百行続くと、読者の集中力は急速に失われます。人間が書いた文章は意識的・無意識的にリズムを変化させていますが、AIはデフォルトでは一定のリズムで文を生成し続けます。

この記事では、52本の記事ドラフトに適用した文末同質化の防止ルールと、プロンプトへの組み込み方を解説します。

## 文末同質化とは何か

文末同質化とは、「文章の終わり方が短時間に繰り返され、単調なリズムが生まれる現象」です。

```
# 文末同質化の典型例

AIを活用することで、開発速度が向上します。
また、品質の安定にも貢献します。
チーム全体の生産性も改善します。
コスト削減にもつながります。
```

4文連続で `〜ます` で終わっています。意味は伝わりますが、読み手に「機械が書いた」という印象を与えます。

人間のライターはこれを本能的に避けます。

```
# リズムが整った例

AIを活用することで、開発速度が向上します。
品質の安定も、副次的な効果の一つ。
チーム全体の生産性改善——これが最大のメリットといえるでしょう。
結果として、コスト削減にもつながります。
```

同じ情報量でも、リズムに変化があることで読みやすさが大幅に向上します。

## なぜAIは文末同質化が起きやすいのか

言語モデルは「次のトークンの確率」を計算して文を生成します。

日本語の文末では `〜す` の後に `。` が来る確率が非常に高く、特にフォーマルな文体を指示すると `〜します。` `〜ます。` が支配的になります。プロンプトで「ですます調で書いてください」と指定するだけでは、この傾向を抑制できません。

また、AIは「直前の数文脈」を参照して次の文を生成するため、同じ文末パターンが続いても「変える理由」を見つけにくいという特性もあります。

## 5つの文末変化ルール

実際にコンテンツパイプラインのプロンプトに組み込んだルールです。

### ルール1: 同じ文末3連続禁止

最もシンプルで効果的なルールです。

```
【NG】
APIにリクエストを送ります。
レスポンスを受け取ります。
データを処理します。

【OK】
APIにリクエストを送ります。
レスポンスを受け取り、データを処理。
以上がリクエスト処理の流れです。
```

「〜ます」が2回続いたら、3回目は別の終わり方を選ぶだけです。このルール1つで文末同質化の大半は解決します。

### ルール2: 体言止めの活用

体言止めは「名詞・名詞句で文を終わらせる」技法です。リズムにアクセントをつける効果があります。

```
【ですます調のみ】
この設計の最大の強みは拡張性です。
追加機能も容易に実装できます。

【体言止めを混ぜる】
この設計の最大の強みは、拡張性。
追加機能の実装も、容易に行えます。
```

体言止めは連続して使わないことが重要です。2〜3文に1回程度が自然なリズムを生みます。

### ルール3: 長短交互リズム

文の長さを意識的に交互に変化させます。

```
【同じ長さが続く例】
Supabaseのリアルタイム機能は、WebSocketを使って実装されています。
RLSを適切に設定することで、セキュリティを担保できます。
また、Edge Functionsを使えばサーバーレスな処理が実現できます。

【長短を交互に配置した例】
Supabaseのリアルタイム機能は、WebSocketを使って実装されています。
RLSも組み合わせる。
この2つの連携で、セキュリティを担保しながらリアルタイム更新を実現できます。
```

短い文が差し込まれることで、読者の目に「休憩点」が生まれます。

### ルール4: 強調は短文断言

重要な主張を伝えたいとき、長い文より短い断言のほうが刺さります。

```
【長い強調文】
この問題を解決するためには、データの取得タイミングとRLSの適用スコープを
正確に理解することが非常に重要であると言えます。

【短文断言】
これが核心です。

RLSは「エラーを返さない」。だからサイレント失敗に気づけない。
```

「これが核心です。」という一文は、次に来る情報への注意を最大化します。

### ルール5: 接続後の文末変化

「しかし」「また」「そのため」などの接続詞の後は、文末を変えるとリズムが整います。

```
【接続後に同じ文末が続く例】
データを取得します。しかし、エラーが発生します。
そのため、リトライ処理を実装します。

【接続後に文末を変えた例】
データを取得します。しかし、エラーが発生する。
そのため、リトライ処理が必要です。
```

## プロンプトへの組み込み方

Claude Code のプロンプトに組み込む場合の実装例です。

```python
WRITING_STYLE_RULES = """
## 文体・リズムルール

以下のルールを必ず守ること:

1. ** 文末3連続禁止 **: 同じ文末（〜ます/〜です/〜する）を3回以上連続させない
2. ** 体言止め活用 **: 3〜5文に1回、体言止めを挟む（連続使用は禁止）
3. ** 長短交互 **: 長い文の後には短い文を配置する（3文以上同じ長さが続かない）
4. ** 断言の短文 **: 重要な主張は「〜。」の短文で独立させる
5. ** 接続語後の変化 **: 「しかし」「また」「そのため」の後は前の文と異なる文末にする

文末バリエーションの例:
- 〜ます / 〜です（敬体の基本）
- 〜する / 〜なる / 〜ある（常体の混入）
- 〜[名詞]。（体言止め）
- 〜でしょう / 〜といえます（推量・やわらかい断言）
- 〜のです / 〜のでした（強調・過去の発見）
"""

def build_article_prompt(topic: str, outline: str) -> str:
    return f"""
あなたはZenn向けの技術記事ライターです。

## 記事の要件
トピック: {topic}
構成: {outline}

{WRITING_STYLE_RULES}

## 出力形式
- Markdownで出力
- です・ます調を基本とする
- 上記の文体ルールを全て適用する
"""
```

## ビフォーアフターの比較

実際の記事ドラフトに適用した例です。

### Before（文末同質化あり）

```
Supabaseのリアルタイムサブスクリプションは、特定のテーブルの変更を
リアルタイムで受け取ることができます。
設定はとても簡単で、数行のコードで実装できます。
チャンネルを作成し、フィルターを設定するだけです。
変更が発生すると、コールバック関数が呼ばれます。
```

`〜ます` → `〜ます` → `〜ます` → `〜ます` の4連続。

### After（ルール適用後）

```
Supabaseのリアルタイムサブスクリプションは、特定のテーブルの変更を
リアルタイムで受け取ることができます。
設定は驚くほど簡単。数行のコードで実装が完了します。
チャンネルを作成し、フィルターを設定する——それだけです。
変更が発生したとき、指定したコールバック関数が自動的に呼ばれます。
```

- 2文目: 体言止め（「簡単。」）+ 短文断言
- 3文目: 常体混入（「設定する」）+ ダッシュによる余韻
- 4文目: 時を表す節の追加でリズム変化

## 52本への4並列適用

52本のドラフトへの一括適用は、Claude Code の並列エージェントを使って実施しました。

```python
# 4並列で文末修正を適用するスクリプトのイメージ
import asyncio
from pathlib import Path

async def fix_article_rhythm(article_path: Path) -> dict:
    """1記事の文末リズムを修正する"""
    content = article_path.read_text(encoding='utf-8')
    
    # Claude APIで文末同質化を修正
    prompt = f"""
    以下の記事の文末リズムを修正してください。
    
    ルール:
    - 同じ文末3連続禁止
    - 3〜5文に1回体言止めを挿入
    - 重要な主張は短文断言に
    - 内容は一切変更しない
    
    記事:
    {content}
    """
    
    # ... API呼び出し ...
    return {"path": article_path, "status": "fixed"}

async def main():
    articles = list(Path("articles").glob("*.md"))
    
    # 4並列で処理
    semaphore = asyncio.Semaphore(4)
    
    async def bounded_fix(path):
        async with semaphore:
            return await fix_article_rhythm(path)
    
    results = await asyncio.gather(*[bounded_fix(a) for a in articles])
    print(f"処理完了: {len(results)}件")

asyncio.run(main())
```

52本を4並列で処理することで、手動修正と比べて大幅な時間短縮。さらに品質が均質になるメリットもありました。

## 日本語特有の考慮点

文末同質化は日本語に特有の問題です。英語では文末の変化が形態的に現れにくいため、同様の問題は起きにくいです。

日本語の文末バリエーションとして使えるのは主に以下のパターン。

| 文末タイプ | 例 | 用途 |
|-----------|-----|------|
| 動詞の丁寧体 | 〜ます / 〜ました | 基本。多用しない |
| 名詞止め | 〜が重要。/ 〜が課題。 | アクセント。2文連続禁止 |
| 形容詞止め | 非常に有効です | 丁寧体との組み合わせ |
| 常体の混入 | 〜する / 〜なる | メリハリをつける |
| 推量 | 〜でしょう / 〜かもしれません | 断定を避けたい場合 |
| 強調疑問 | 〜ではないでしょうか | 読者への問いかけ |

これらを意識的に組み合わせることで、リズムに変化が生まれます。

## まとめ

AI生成記事の文末同質化を防ぐ5ルール。

1. ** 同じ文末3連続禁止 ** — これだけで大半の問題を解決
2. ** 体言止め活用 ** — 3〜5文に1回のアクセント
3. ** 長短交互リズム ** — 読者に「休憩点」を与える
4. ** 強調は短文断言 ** — 重要な主張は短く独立させる
5. ** 接続後の文末変化 ** — 接続語の後は文末を変える

これらをプロンプトに明示的に組み込むことで、AIの生成する記事品質が大幅に改善します。内容の正確性だけでなく、読み手の体験にまで目を向けることが、AI生成コンテンツの次の課題といえます。

文末同質化の改善は、AI生成コンテンツと人間の書いた文章の距離を縮める、地味ながら効果的な手法の一つです。
