## 概要

8MB超のYouTube転写テキスト（9チャンネル223本）から、重要な知見を効率よく抽出しなければならない状況に直面した。手作業や逐次処理では現実的な時間で終わらない。

そこで Claude Code の **Task tool（サブエージェント）** を使い、sonnet モデルを7本同時に走らせた。結果、約30分で49件の重大知見を抽出し、優先度テーブルにまとめるところまで完了した。逐次実行と比較すると約7倍のコスト効率。

本記事はこの実践の記録。コードと設計の考え方を具体的に共有する。

:::message
本記事で扱うのは **Task tool（サブエージェント）** による並列実行。2026年2月リリースの Agent Teams（チーム間通信あり）とは別機能です。Task tool は Claude Code の標準機能で、今日から使えます。
:::

## 背景：なぜ8MBの転写テキストが存在したか

PPP/PFI・地域創生分野の事業計画策定に向けて、国交省・文科省・NILIM（土木研究所）・地方創生推進室などの公開YouTubeチャンネルを体系的に調査していた。

Whisper で転写した結果は次のとおり。

| チャンネル | 本数 | 主なテーマ |
|------------|------|------------|
| PFS/SIB関連 | 6本 | 成果連動型民間委託 |
| 福岡市関連 | 5本 | 自治体PPP先行事例 |
| NILIM | 11本 | 公共施設・インフラ研究 |
| 文科省 | 29本 | 廃校・教育施設活用 |
| 国交省(MLIT) | 50本 | PFI・コンセッション |
| 東洋大学 | 49本 | PPP/PFI学術講座 |
| 地方創生 | 63本 | 人口・企業版ふるさと納税 |

合計223本、転写テキストは 8MB超。これを1本ずつ読んでいたら数日かかる。

## 設計思想：なぜ完全並列が成立したか

サブエージェント並列化が効果を発揮するのは **タスク間の依存関係がゼロのとき** だ。

今回のケースは理想的だった。各チャンネルが扱うドメインは独立している。PFS/SIBの知見抽出は国交省の転写を参照しない。文科省の廃校データは地方創生の企業版ふるさと納税とは別のファイルに入っている。

依存関係がないなら、並列数をそのままスループットの倍率にできる。

```
逐次実行: 7チャンネル × 推定60分 = 約7時間
並列実行: max(各チャンネルの処理時間) ≈ 30分
```

ただし「依存がないから並列すればいい」では設計として浅い。各サブエージェントに渡すプロンプトの設計と、統合エージェント（opus）への出力フォーマット統一が品質を左右する。

## 実装：Task tool でサブエージェントを並列起動する

### 基本構造

Claude Code の **Task tool** は、サブエージェントを独立したコンテキストで起動し、その結果を呼び出し元に返す仕組み。複数の Task をほぼ同時に発行すれば、並列実行に近い効果が得られる。

```
親エージェント（orchestrator）
├── Task: PFS/SIB転写 → sonnet で知見抽出
├── Task: 福岡市転写  → sonnet で知見抽出
├── Task: NILIM転写   → sonnet で知見抽出
├── Task: 文科省転写  → sonnet で知見抽出
├── Task: MLIT転写    → sonnet で知見抽出
├── Task: 東洋大転写  → sonnet で知見抽出
└── Task: 地方創生転写 → sonnet で知見抽出
                           ↓（全完了後）
             opus が統合・優先度テーブル作成
```

### プロンプト設計

各サブエージェントへの指示文は統一フォーマットにした。抽出精度が上がり、後段の統合が楽になる。

```markdown
## 役割
あなたはPPP/PFI・公共施設活用分野の専門アナリストです。
以下の転写テキストを読み、重大知見を構造化してください。

## 対象ファイル
{channel_name}/transcripts/*.txt（合計{file_count}本）

## 出力フォーマット（必須）
各知見を以下の形式で出力してください：

### KN-{番号}: {知見タイトル}（1行）
- **カテゴリ**: [制度/数値/事例/リスク/機会] から選択
- **信頼度**: [高/中/低]（一次ソースがYouTube内で明示されているか）
- **概要**: 3行以内。数値があれば必ず含める
- **事業計画への示唆**: 1行

## 抽出基準
- 「高」優先: 具体的数値付き・政策の方向性を示す発言・実際の事業スキーム
- 「中」優先: 専門家見解・比較事例・課題指摘
- 「低」: 一般論・重複情報は除外

## 注意事項
- 転写の誤字は文脈から補正して読む
- 固有名詞（自治体名・法人名）は転写のまま記載
- 1チャンネルにつき5〜10件が目安（品質 > 量）
```

このプロンプトで重要なのは3点。

1. **出力フォーマットを厳格に指定する**: 後段で統合するとき、フォーマットが揃っていないと処理コストが跳ね上がる
2. **信頼度を必須項目にする**: 転写テキストは誤認識が多い。話者の主観と事実の区別もつきにくい。信頼度ラベルをつけることで統合時の優先度判断が楽になる
3. **抽出件数を「目安」として示す**: 上限を設けないと粗い知見が大量に出てくる。「5〜10件」という制約が質を上げる

### 統合エージェント（opus）のプロンプト

サブエージェント7本の出力が揃ったら、opus に統合と優先度付けを依頼した。

```markdown
以下は7チャンネルの転写テキストから抽出された知見リスト（計{n}件）です。

## あなたのタスク
1. 重複・類似知見を統合（事業計画への示唆が同一のものをマージ）
2. 全知見に4段階優先度を付与
   - 🔴S: 即時に事業計画へ反映すべき（競争優位・リスク回避に直結）
   - 🟠A: 今期中に反映（市場理解・事業スキーム精緻化に有用）
   - 🟡B: 中期で検討（将来の事業拡張・付加価値に関連）
   - 🟢C: 参考情報（背景理解・競合分析用）
3. 優先度テーブルとして出力

## 優先度判断基準
- S: 数値が具体的、かつ現在の事業計画に「穴」があることを示している
- A: 事業スキームの選択肢を広げる・競合の動向を示す
- B/C: 長期的に有用だが今すぐ反映しなくてよい

## 出力形式
| 優先度 | 知見ID | タイトル | チャンネル | 事業計画への示唆 |
|--------|--------|---------|-----------|----------------|
| 🔴S | KN-01 | ... | NILIM | ... |
...

テーブルの後に、S・A優先の知見について詳細解説を追記してください。
```

opus を統合に使う理由は、優先度の **判断品質** が問われるから。抽出（情報の読み取り）は sonnet で十分だが、「この知見が事業計画のどの穴を埋めるか」という判断には推論の深さが必要になる。

## 実行結果：49件の知見と優先度テーブル

### チャンネル別抽出件数

| チャンネル | 抽出件数 | 処理時間（推定） |
|------------|----------|-----------------|
| PFS/SIB | 9件 | 約4分 |
| 福岡市 | 7件 | 約3分 |
| NILIM | 8件 | 約5分 |
| 文科省 | 5件 | 約3分 |
| MLIT | 7件 | 約6分 |
| 東洋大 | 6件 | 約5分 |
| 地方創生 | 7件 | 約4分 |
| 合計 | 49件 | 約30分（並列） |

逐次実行の場合、MLIT（50本）だけで推定1時間以上かかる。7チャンネルを通せば7時間超。並列化で約14分の1に圧縮した計算になる。

### 優先度テーブル（抜粋）

統合後の優先度分布は 🔴S:5件 / 🟠A:11件 / 🟡B:9件 / 🟢C:5件（重複統合後の実数）。

🔴S に分類された5件から3件を紹介する。

**KN-03: PFS普及率1割未満 + 20万人未満自治体の9割がPFI未実施**

国内のPFS（成果連動型民間委託）導入実績は全体の1割未満にとどまる。PFIについては、人口20万人未満の自治体の約9割が一度も導入していない。「市場はまだ整備段階」という自社ポジションの根拠になる数値。

**KN-07: 上下水道の行政一体化 + 老朽管路の耐震化率9%**

国交省データ（NILIM）によると、全国の水道管路のうち耐震化が完了しているのは9%。上下水道を統合した広域PPPのニーズは急拡大している。事業計画の §2-1c に即時反映すべき数値。

**KN-22: 企業版ふるさと納税の年間実績631億円・人材派遣型の拡大**

企業版ふるさと納税の2024年度実績は631億円。うち「人材派遣型」（社員を自治体に派遣する形態）が急増。資金だけでなく人材が動く仕組みとして事業計画に組み込める。

## 実際に事業計画へ反映した16項目

知見抽出は抽出で終わらせなかった。同じセッション内で、🔴S5件 + 🟠A2件の計7件を business-plan-v5.3 に即時反映した。

```
§2-1b: PFS普及率1割未満 + 20万人未満自治体の9割PFI未実施（KN-03）
§2-1c: 上下水道行政一体化 + 耐震化率9%（KN-07）
§2-1d: 公共施設更新費の4倍財政ギャップ（KN-12）
§6-8:  廃校スモールコンセッション標準モデル（KN-18）
§6-9:  WTP算定・ロジックモデル設計支援（KN-21）
§6-10: 企業版ふるさと納税 人材・資金スキーム（KN-22）
§10-3: 首長会議56自治体アプローチ戦略（KN-31）
```

転写テキストを読み始めてから事業計画への反映完了まで、同一セッション内で約1.5時間。その後、抽出した知見を元に ISVD サイトの labs 記事3本を品質パイプライン通過でコミットするところまで4時間で完走した。

## 設計上の注意点と失敗パターン

実際に動かして見えてきた注意点を3つ共有する。

### 注意点1：サブエージェントのファイルアクセスパスを絶対パスで渡す

Task tool のサブエージェントは、呼び出し元と異なるコンテキストで動く。相対パスを渡すと「ファイルが見つからない」エラーが頻発する。

```markdown
# NG（相対パス）
対象ファイル: transcripts/nilim/*.txt

# OK（絶対パス）
対象ファイル: /Users/naoya/dev/projects/public-asset/data/nilim/transcripts/*.txt
```

### 注意点2：出力フォーマットの「型崩れ」に備える

7本のサブエージェントが返してきた出力は、プロンプトで指定したフォーマットと微妙にずれることがある。知見番号の付け方が「KN-1」と「KN-01」で混在したり、テーブルの列数が違ったりした。

統合エージェントへの指示に「型崩れがあっても内容を優先して統合せよ」と明示しておくと、opus が柔軟に吸収してくれる。

### 注意点3：信頼度「低」の知見は統合に含めない

転写テキストはノイズが多い。Whisper の認識ミス、講演者の個人的な見解、文脈の切れ目での誤解釈。信頼度「低」の知見を統合に含めると、優先度テーブルが汚染される。

サブエージェントへのプロンプトで「信頼度低は抽出しない」と明示するより、統合エージェントへの指示で「信頼度低は除外して統合」と伝えるほうが結果が安定した。サブエージェント段階では情報を失わず持ってきて、統合でフィルタリングする設計。

## コストと所要時間の実測

| 指標 | 逐次実行（推定） | 並列実行（実測） |
|------|----------------|----------------|
| 所要時間 | 約7時間 | 約30分 |
| モデル | opus 単体 | sonnet×7 + opus×1（統合） |
| 抽出精度 | 理論上は同等 | 優先度判断は opus が担保 |
| コスト効率 | 基準 | 約7倍（処理時間あたり） |

sonnet を抽出に、opus を統合・判断に使い分けた。「抽出は情報の読み取りなので sonnet で十分」という判断は正しかった。opus の逐次レビューより、sonnet の並列抽出 + opus の統合のほうが、品質を下げずにスピードを大幅に上げられる。

## どんな場面で使えるか

今回は YouTube 転写テキストだったが、同じパターンが使える場面は多い。

- **法改正や行政通知の大量処理**: 省令・告示・Q&Aが大量にあるとき、条文ごとにサブエージェントを立てて影響箇所を抽出
- **競合調査**: 複数サービスのドキュメント・ブログ・SNS投稿を並列で読ませて比較テーブルを作る
- **インタビュー記録の分析**: 複数のインタビュー書き起こしから、テーマ別の発言を並列抽出
- **コードベースのドメイン別レビュー**: モジュールごとにサブエージェントを立てて、セキュリティ・パフォーマンス・可読性をチェック

共通する条件は「タスク間の依存がなく、入力データが分割可能で、出力フォーマットを揃えられる」こと。この3条件が揃えば、並列化は効く。

## まとめ

今回の実践で確認できたことを整理する。

**やったこと**

- YouTube 9チャンネル 223本 / 8MB超の転写テキストを sonnet×7 で並列処理
- 約30分で49件の重大知見を抽出し、opus が統合優先度テーブルを作成
- 同日セッション内で事業計画 v5.3 に16項目反映、サイト記事3本をコミット

**設計の要点**


1. 並列化は「依存関係がゼロのタスク群」に限定する
2. サブエージェントへの出力フォーマット指定を厳格にする（統合コストを下げる）
3. 抽出（情報読み取り）は sonnet、判断（優先度付け）は opus の役割分担
4. 信頼度ラベルを抽出段階から付与し、統合時にフィルタリングする

Claude Code の Task tool は、使い慣れると「大量の非構造化テキストを構造化情報に変換するパイプライン」として機能する。今回はYouTube転写だったが、汎用性は高い。大量テキストの処理に悩んでいる方は試してみてほしい。

### 統合エージェント用（opus）

```markdown
以下は{N}チャンネルの転写から抽出された知見（計{M}件）です。

## タスク
1. 重複・類似知見を統合
2. 全知見に優先度を付与（🔴S/🟠A/🟡B/🟢C）

- B/C: 中長期・参考情報

## 統合方針
- 型崩れがあっても内容を優先して統合
- 信頼度「低」の知見は除外して統合
```
